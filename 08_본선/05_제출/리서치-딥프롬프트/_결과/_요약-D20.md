# D20 요약 — 금융 AX의 답은 만능 LLM이 아니라 에이전트 운영체제다

## ⚡ 30초 핵심
금융 AX에서 LLM은 신용평가·FDS·문서추출·조기경보의 주 모델이 아니다.  
LLM은 특화모델, 규칙엔진, 레거시 어댑터, 승인 게이트, 감사 로그를 호출하고 묶는 오케스트레이터다.  
JB LocalGuard OS는 “AI 챗봇”보다 “금융 업무 실행 하네스”로 밀어야 설득력이 생긴다.

## 무엇을 알게 됐나
- 신용평가·여신심사에는 XGBoost, LightGBM, Random Forest 같은 트리 계열이 여전히 강하다. 표 데이터와 시계열 예측에서 LLM은 일관된 우위를 보이지 못했고 비용도 크다.
- FDS와 이상징후 탐지는 규칙엔진, GBM, GNN, 시퀀스 모델의 조합이 맞다. 사기·세탁은 거래 관계와 이벤트 흐름을 봐야 해서 LLM 단독 구조가 약하다.
- 문서 IE·OCR은 LayoutLMv3와 Donut 같은 Document AI가 맞다. LayoutLMv3는 FUNSD F1 92.08, Donut은 CORD F1 91.6·Acc 93.5와 1.2초/건을 제시했다.
- 금융 AI의 성패는 모델보다 레거시 접합 방식에 걸린다. 카카오뱅크 담보심사 사례는 비동기 작업 체인, 상태 기반 대시보드, 기존 계정계 프레임워크 재사용을 보여준다.
- 빅뱅 전환은 위험하다. TSB의 2018년 IT migration 실패는 business-as-usual 복귀가 2018년 12월까지 밀렸고, 비용은 2024년 가격 기준 약 4억 1,216만 파운드로 제시됐다.
- MCP는 외부 시스템 연결 표준이지만 보안 완성품은 아니다. MCP authorization은 optional이고, 기업은 하네스에서 권한, 샌드박스, 감사, 허용목록을 강제해야 한다.
- 같은 모델도 하네스와 scaffolding에 따라 성능이 달라진다. ReAct, Toolformer, DSPy, Anthropic SWE-bench 근거는 “프롬프트”보다 실행 환경 설계가 성능을 좌우한다고 말한다.

## 제품·심사에 어떻게 쓰나
- JB LocalGuard OS의 Skill 층을 “LLM 기능 묶음”이 아니라 운영계약 실행 하네스로 정의한다. Case → Intake → Evidence → Decision Support → Action → Audit 흐름을 제품 구조로 보여준다.
- Score Skill을 별도 계층으로 둔다. 신용은 XGBoost/GBM, 이상거래는 규칙+GNN/시퀀스, 문서는 LayoutLMv3/Donut, 추천은 ranking 모델을 호출하고 점수·확률·특징기여·모델버전을 구조화해 반환한다.
- Legacy Adapter Skill을 강조한다. 코어뱅킹·정보계·외부기관 연동은 표준 계약으로 감싸고, 내부는 Deferred Batch, 상태 전이, 대시보드로 운영한다.
- Tool Registry와 Policy Engine을 전면에 둔다. MCP, CLI, 내부 REST, 배치 잡, 검색 서버마다 테넌트, 데이터 등급, 승인 필요 여부, 허용 파라미터를 계약화한다.
- 심사 설득 지표는 “모델 정확도”보다 하네스 전후 pass rate, tool success rate, approval burden, time-to-resolution, legacy error rate로 잡는다.

## ⚠️ 주의·한계
- 한국 은행의 FDS·신용평가·추천 운영 성능 수치는 공개 1차 자료가 부족하다. 실제 AUC, 탐지율, 오탐률은 대체로 비공개다.
- 벤더나 기업 블로그의 “N배 개선”, “정확도 99%”, “시간 98% 절감”은 2차·미검증으로 격리해야 한다.
- ₩ON, BC Card Finance LLM Benchmark, KR-FinBert 계열은 존재 확인 수준이다. 모델카드와 GitHub README를 현업 운영 품질로 비약하면 안 된다.
- MCP는 빠르게 성숙 중인 신흥 표준이다. 연결 표준의 성숙과 금융 운영 보안 통제는 같은 말이 아니다.
