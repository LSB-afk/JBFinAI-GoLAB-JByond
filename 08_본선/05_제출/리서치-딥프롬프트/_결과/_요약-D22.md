# D22 요약 — 금융권 AI 도입은 로컬 우선과 통제 승격이 답이다

## ⚡ 30초 핵심
은행이 외부 LLM을 꺼리는 이유는 보수성이 아니라 고객정보 반출, 제3자 IT 리스크, 감사 책임 때문이다.  
해법은 “외부 금지”가 아니라 민감 업무는 로컬, 공개·비식별·고난도 업무만 외부 premium으로 승격하는 구조다.  
JB LocalGuard OS는 이 구조를 PII 비반출, 정책엔진, 로그, 사람 승인으로 보여 주면 된다.

## 무엇을 알게 됐나
- 금융당국은 2024년부터 생성형 AI 활용을 열었지만, 방향은 무제한 허용이 아니다. “상용 AI + 내부망 오픈소스 AI”의 투트랙이다.
- 외부 API는 기본 무학습이어도 은행 기준에선 여전히 외부 전송이다. OpenAI API도 남용 모니터링 로그를 최대 30일 보관할 수 있다고 밝힌다.
- 모델 선택 기준은 벤치마크 1등이 아니다. 상업 라이선스, 로컬 호스팅, 공급망 리스크, 한국어 지원, 감사 가능한 운영 구조를 같이 봐야 한다.
- 국산 후보는 NAVER HyperCLOVA X SEED와 Upstage Solar Open이다. EXAONE 3.5 공개본은 연구용 성격이라 은행 실서비스에는 바로 쓰기 어렵다.
- 로컬 LLM의 비용 변수는 전기료보다 GPU 활용률과 모델 크기다. 4xH100 + 70B 추정 월 TCO는 1,000만원, 50% 활용 월 처리량은 약 5.02억 토큰이다.
- GPT-4.1 mini 같은 저가 API와 로컬은 가격 경쟁이 어렵다. 4xH100 70B가 GPT-4.1 mini보다 싸지려면 월 약 97억 토큰이 필요하다.
- 작은 로컬 모델도 하네스가 좋으면 쓸모가 크다. GAIA의 인간 92% vs GPT-4+plugins 15%, mini-SWE-agent의 SWE-bench Verified 65%는 모델보다 도구·검증 루프가 성능을 좌우한다는 증거다.

## 제품·심사에 어떻게 쓰나
- 데모 메시지는 “외부 LLM 안 씁니다”가 아니다. “민감데이터는 외부로 안 나가고, 필요할 때만 통제 승격합니다”가 더 세다.
- 고객식별정보, 상담기록, 내부 규정, 상품문서는 로컬 경로로만 보내라. 외부 호출은 정책엔진이 차단하거나 승인 대기시키는 장면을 보여 주면 된다.
- 같은 하네스에 Codex, Claude, 로컬 모델을 꽂되 정책과 로그는 유지된다는 점을 보여 줘라. 모델 교체 가능성이 곧 주권과 감사 가능성이다.
- 로컬 8B~32B 모델의 역할은 규정 QA, 상담 스크립트 초안, 상품 비교표, 내부 API 호출 같은 반복 업무다. “전부 다 잘함”보다 “반복업무를 안정적으로 잘함”이 설득력 있다.
- 비용 슬라이드는 단순하게 간다. 대량·반복·민감 업무는 로컬, 희소·고난도 업무는 외부가 맞다는 곡선을 숫자로 보여 주면 된다.

## ⚠️ 주의·한계
- 한국 은행의 내부 보안정책 원문은 대부분 비공개다. 국내 은행 사례 일부는 보도자료·언론·벤더 발표를 통한 간접 확인이다.
- 온프레 GPU 구매 단가와 로컬 TCO는 [추정] 시나리오다. 실제 조달가, 목표 동시성, 운영 인력을 넣어 다시 계산해야 한다.
- 모델·가격·라이선스는 변동이 빠르다. 원문 가격 비교는 2026-07-01 접속 기준이다.
- DeepSeek·Qwen의 호스티드 서비스 리스크와 셀프호스팅 가중치 리스크를 섞으면 안 된다. 내부 정본 `_canon.md`, `D5b`, `D14`, `D19`와 충돌하면 내부 정본이 우선이다.
