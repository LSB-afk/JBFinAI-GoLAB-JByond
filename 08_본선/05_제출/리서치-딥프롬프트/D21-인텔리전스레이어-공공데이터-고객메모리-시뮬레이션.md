# D21 — 인텔리전스 레이어: 정부·공공 빅데이터 · 통합 고객 메모리 · 위험예측·시뮬레이션(월드모델)

## 메타 (프롬프트 밖 — 생성·운영용)
- **도메인**: 에이전트가 **무엇을 먹고 똑똑해지나**. (A) 정부 정책·실시간 데이터·뉴스·공공 빅데이터 활용, (B) **통합 고객 메모리**(채널·부서·계열사 횡단으로 한 고객 데이터가 통합되고 상담할수록 고도화), (C) **위험예측·시나리오 시뮬레이션·월드모델**(행동 전에 가상 시뮬레이션 → 괜찮으면 현실 재현).
- **티어**: ★★★
- **심사 항목**: 3(AI Agent 설계 — 인텔리전스·예측 능력) · 5(혁신·확장·운영리스크 — 통합 고객지능·시뮬레이션이 만드는 차별·확장)
- **실행 배치**: 배치 3 (종합·차별 — D7a·D11·D17 인용)
- **경계 메모**:
  - **D21 ↔ D7a**: 공공·금융 데이터 *소스·API·라이선스 목록* = D7a / 그것을 *정책추적·뉴스·실시간 인텔리전스로 활용·가공*하는 설계 = D21.
  - **D21 ↔ D11**: 메모리의 *학술 구조(episodic/CBR)* = D11 / **고객 단위 통합 메모리의 제품·데이터통합 능력(360·횡단·고도화 루프)** = D21.
  - **D21 ↔ D17**: 계열사 데이터 공동이용의 *규제·클린룸 근거* = D17 / 통합 고객 메모리가 만드는 *제품 가치·인텔리전스* = D21. (횡단 통합의 합법성은 D17이 보증, D21은 그 위의 활용.)
  - **D21 ↔ D14**: 데이터 품질·편향·공정대출 = D14 / D21은 예측·시뮬레이션의 *활용 설계*.
- **제품 연결**: JB LocalGuard OS가 단순 응답봇이 아니라 **① 공공·정책·뉴스를 실시간 흡수해 신호화**하고, **② 한 고객의 데이터를 채널·부서·계열사 횡단으로 통합해 상담할수록 고도화되는 살아있는 고객지능**을 운용하며, **③ 행동 전에 가상 시나리오를 시뮬레이션(월드모델)해 선택지·리스크를 제시**하는 "인텔리전스 OS"임을 정당화. 예: 딸기 농가 여신 — 작황·기상·시세 데이터로 먼저 시뮬레이션 → "괜찮다" 판정 시 실제 행동안 상신. 운영계약상 Case에 **공공신호·고객메모리·시뮬레이션 결과를 Evidence로 누적**.

## 딥리서치 프롬프트 (이 아래 통째로 붙여넣기)

당신은 **데이터·AI 인텔리전스 시스템 리서처**다. 목표는 금융 AI 에이전트가 "응답"을 넘어 **외부 세계(정부·공공·뉴스)를 흡수하고, 한 고객을 횡단 통합해 점점 더 잘 이해하며, 행동 전에 가상 시뮬레이션으로 리스크를 검증"하는 세 인텔리전스 능력**을, **공개 1차 자료·동료평가 연구·실제 사례**로 입증·구조화하는 것이다. 결과는 거버넌드 AI 에이전트 콘솔이 "지능형 운영 레이어"임을 정당화하는 데 쓰인다.

### 알아내야 할 것 (핵심 질문)

**A. 정부·공공 빅데이터 · 정책 · 실시간 · 뉴스 인텔리전스 (활용)**
1. 한국 금융이 **정부 정책·규제 변화를 실시간 추적·반영**하는 방식: 정책/고시/보도자료/공공 알림을 모니터링→신호화하는 패턴, 정책 변화가 여신·소상공인·전세·사기 대응에 미치는 트리거. (데이터 *소스·API 목록*은 D7a — 여기선 *추적·신호화·활용*.)
2. **뉴스·이벤트 추적(news/event intelligence)**을 금융 리스크·고객 대응에 쓰는 사례·기법: 실시간 뉴스 스트림, 지역·업종 이벤트 감지, 조기경보. NLP 이벤트 추출·정책 모니터링의 대표 방법.
3. **공공 빅데이터를 활용한 금융·정책 분석** 사례: 정부가 제공한 빅데이터(상권·고용·부동산·소상공인 등)로 위험·기회를 포착한 국내외 사례와 효과.

**B. 통합 고객 메모리 — Customer 360 · 횡단 통합 · 고도화 루프 (운용)**
4. **단일고객뷰(Customer 360)·고객 데이터 통합**: 전화상담·창구·앱·여러 부서·여러 계열사에 흩어진 한 고객의 데이터를 **하나로 통합**하는 아키텍처(CDP·MDM·고객 식별·엔티티 해소)와 금융권 사례. (계열사 횡단 *공동이용의 합법성·클린룸*은 D17 — 여기선 *통합이 만드는 능력*.)
5. **상호작용할수록 고도화되는 고객 메모리(longitudinal/interaction-driven personalization)**: 상담·접점이 쌓일수록 고객 이해가 깊어지는 설계(누적 프로파일·선호·맥락), 그리고 **프라이버시 보존 개인화**(민감정보 누적의 통제). 효과와 위험을 함께.
6. **횡단 통합 고객지능의 리스크**: 잘못된 통합·오식별·확증편향·프라이버시·신용정보 목적외 이용. 알려진 실패·완화책. (최소 1건 의무.)

**C. 위험예측 · 시나리오 시뮬레이션 · 월드모델 (행동 전 검증)**
7. **금융 위험예측·조기경보**를 넘어 **시나리오/스트레스 시뮬레이션·디지털 트윈·월드모델**을 의사결정 전에 돌리는 패러다임: "행동을 현실에 적용하기 전 가상 환경에서 먼저 시뮬레이션→안전하면 실행"하는 설계(simulation-before-action, what-if, agentic planning with world models). 대표 연구·금융/산업 적용 사례.
8. **시뮬레이션 기반 의사결정 지원**의 구체 패턴: 여신·농가·소상공인처럼 외부 변수(기상·작황·시세·경기)가 큰 케이스에서 다변수 시뮬레이션으로 선택지·리스크 분포를 제시하는 방법. 몬테카를로/시나리오 트리/디지털 트윈/강화학습 플래닝 중 무엇이 적합한가.
9. **시뮬레이션의 한계·오용**: 모델 가정 오류·과신·"시뮬레이션이 현실을 보장하지 않음"·데이터 편향 전파. 완화책. (최소 1건 의무.)

**D. (범용 덤 유도)** 위 외에 "에이전트가 똑똑해지는 인텔리전스 능력"으로 금융 AX에서 차별을 만드는 것(예: 지식그래프 기반 추론, 외부 시그널 융합)이 있으면 추가 카테고리화하라.

### 출처 규칙 (반드시 준수)
- **공통 5가드**: ① 발견마다 **출처 URL + 원문 짧은 인용 + 발행처·날짜 + 신뢰도(1차/2차/[미검증])**. ② 핵심 수치·효과 **2출처 교차**. ③ **링크 미접속 시 인용 금지.** ④ **한국 맥락·최신(2023~2026) 우선.** ⑤ 의심 시 보수적 [미검증].
- **도메인 가드**: **"고객 데이터 통합/개인화 효과"·"예측 정확도" 수치는 벤더 사례 = 2차**, 1차 = 논문·공식 발표·규제기관 자료. **신용정보·개인정보 목적외 이용·횡단 통합의 합법 범위는 단정 금지** — "규제상 가능 여부는 D17/D5a가 판정, 여기선 기술 능력만"으로 한정하고 추정에 [미검증]. 월드모델·디지털트윈은 **동료평가/공식 1차** 우선(블로그 과장 [미검증]).
- **SSOT 우선**: 내부 정본(`_canon.md`)·D7a(데이터 소스)·D17(공동이용 규제)와 충돌 시 그쪽이 승리.

### 한국 맥락·기간
- 한국 금융·공공데이터·정책 추적 사례 최우선, 글로벌 보강. 직접 사례 희소 시 명시.
- 기간: 월드모델·LLM 시뮬레이션·고객지능은 **2023~2026 우선**, 시뮬레이션·예측 고전 기법은 연도 명시 허용.

### 출력 형식 (이 구조로 답하라)
1. **핵심 발견 요약** — A(공공·정책·뉴스)·B(통합 고객 메모리)·C(시뮬레이션·월드모델)·D(덤) 블록별.

8. **갭 / [미검증]** — 한국 직접 사례 희소성, 합법 범위 미확정(D17/D5a로 이관), 검증 못 한 효과 수치.

### 딥리서치 깊이 지시 (Deep Research)
- 세 능력을 **"외부신호 흡수 → 고객 통합지능 → 행동 전 시뮬레이션"의 의사결정 파이프라인**으로 엮어 인과를 추적하라. 단순 기능 나열 금지.
- **"행동을 현실에 적용하기 전 시뮬레이션으로 먼저 검증"** 패러다임(simulation-before-action / world model)의 1차 근거를 우선 파라 — 우리 핵심 차별이다.
- **횡단 고객 통합의 합법 범위는 단정하지 말고** "기술적으로 가능 vs 규제상 허용(D17/D5a 판정)"을 분리해 표기하라.
- 효과 수치는 출처·산정방식 필수, 벤더 사례는 [2차]. 한계 표(B·C 각 1건) 비우지 말 것.
- 데이터 소스·API 목록=D7a, 공동이용 규제=D17, 데이터 품질·공정성=D14, 법조문=D5a — **경계에서 멈추고 참조 표기.**
- 제출 전 자체 점검: "세 능력이 파이프라인으로 엮였는가 / 합법-가능을 분리했는가 / 시뮬레이션·월드모델 근거가 1차인가 / 한계 B·C 각 1건인가."

## Opus 점검 로그 (2패스)

### 패스 1 — 적대적 비판 → 보완
- **비판 ① D7a·D11·D17과 정면 중복.** 데이터·메모리·계열사 통합이 셋과 겹쳐 SSOT 충돌·중복.
  - **보완**: 메타에 4경계 규칙을 명시 — *소스 목록=D7a / 학술 메모리=D11 / 공동이용 규제=D17 / 품질·공정성=D14*. D21 고유 축 = *활용·통합지능·시뮬레이션*으로 좁히고, 본문 곳곳에 "경계에서 멈추고 참조".
- **비판 ② 횡단 고객 통합의 합법성 단정 위험(규제 지뢰).** 신용정보·개인정보 목적외 이용을 "가능"으로 단정하면 심사·컴플 반론.
  - **보완**: 도메인 가드·깊이지시에 "기술적 가능 vs 규제상 허용(D17/D5a 판정)을 **분리 표기**, 추정 [미검증]"을 강제. 출력 7에 "합법 범위는 D17/D5a로 이관"을 명시.
- **비판 ③ 월드모델·시뮬레이션이 공상으로 흐를 위험.** 멋진 개념만 모으면 핍진성 훼손.
  - **보완**: C7~C9에 동료평가/공식 1차 우선·"시뮬레이션이 현실을 보장하지 않음" 한계 의무·구체 기법(MC/디지털트윈/RL플래닝) 비교를 박고, 딸기 농가 같은 **구체 금융 케이스**로 스코프 고정.

### 패스 2 — 재비판 → 보완
- **재비판 ① '상담할수록 고도화'가 측정 불가 슬로건.** 효과 근거 없이 주장하면 약함.
  - **보완**: B5를 longitudinal/interaction-driven personalization의 *설계+효과+프라이버시 통제*로 구체화하고 출력 3 표에 "효과·리스크" 열, 효과 수치 2출처 교차 요구. (효과의 *정량 KPI*는 D13·D8 참조로 경계.)
- **재비판 ② 정책·뉴스 추적이 D7a 소스 나열로 회귀.** 또 소스 목록만 나오면 중복.
  - **보완**: A1~A3을 "소스"가 아니라 **"추적→신호화→금융 대응 트리거"의 활용 흐름**으로 못 박고 출력 2 표에 "대응 트리거" 열을 요구. 소스는 D7a 참조로 명시.
- **재비판 ③ 세 블록이 따로 놀 위험.** A·B·C가 분절되면 "기능 모음"으로 보여 차별 약화.
