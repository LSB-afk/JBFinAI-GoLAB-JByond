# D12 요약 — 한국 금융권 AI RM 콘솔 규제·인증 통과 로드맵

## ⚡ 30초 핵심
외부 퍼블릭 LLM을 은행 내부망에서 쓰려면 기술보다 먼저 제도 경로를 열어야 한다. 순서는 `온프레미스 토큰화 PoC -> 혁신금융서비스 지정 -> 금융보안원 클라우드 안전성 평가 -> ISMS-P 범위 반영`이 맞다. 고객 PII 직접 반출은 안 되고, 가명정보 수준까지 허용된다는 전제를 깔아야 한다.

## 🎯 바로 쓸 것 (에이전트·제품 즉시 활용)
- 제품 제안서와 아키텍처 문서에 `전자금융감독규정 제15조 제1항 제5호 예외 특례` 확보 필요를 명시한다. 내부망에서 외부 LLM API 호출은 혁신금융서비스 지정 없이는 어렵다.
- PoC 1단계는 온프레미스 토큰화 엔진으로 고정한다. 이름, 계좌번호, 주민번호를 로컬에서 `[PT_ID_9812]` 같은 토큰으로 바꾸고 매핑 테이블은 내부 보안영역에만 둔다.
- 클라우드 안전성 평가는 CSP의 `CSAP`, `MTCS`, `CSA STAR` 보유 여부부터 확인한다. 있으면 기본보호조치 109개를 생략하고 금융 특화 54개 항목 중심으로 줄일 수 있다.
- ISMS-P는 제품 출시 직전이 아니라 지금 준비를 잡는다. 원문 기준 준비 포함 최소 5~8개월, 심사 수수료 600만~1,100만 원, 매년 사후심사, 유효기간 3년이다.
- CSAP이 필요한 SaaS 확장 계획이면 공급사 주도로 따로 가져간다. 최근 기준 평균 2개월, 중소기업은 수수료 80% 지원, 사후 평가는 무료 서면평가로 바뀌었다.
- 샌드박스 신청서에는 2024-11-27, 2024-12-11 생성형 AI 내부망 이용 지정 사례를 벤치마크로 넣고, 보안 계획서와 데이터 흐름도를 같이 낸다.

## 🙋 전문가에게 묻는다면
- **Q. 왜 샌드박스가 먼저인가?** A. 물리적 망분리 예외가 필요하기 때문이다. 외부 LLM API 연동의 적법 경로가 여기서 열린다.
- **Q. 가명처리만 하면 바로 외부 LLM에 보낼 수 있나?** A. 아니다. 로컬 토큰화 엔진, 반출 통제, 클라우드 안전성 평가까지 같이 붙어야 한다.
- **Q. 금융보안원 평가는 누가 책임지나?** A. CSP가 아니라 도입 금융회사다. 중요도 평가와 사후 보고 책임도 금융회사 쪽이다.
- **Q. ISMS-P는 필수인가?** A. 금융회사엔 사실상 기본이고, SaaS 공급사도 입찰과 보안성 검토를 통과하려면 거의 필수에 가깝다.
- **Q. 지정대리인이나 위탁테스트로 갈 수는 없나?** A. 가능은 하지만 목적이 다르다. 외부 LLM 내부망 연동 특례를 직접 여는 수단은 혁신금융서비스가 더 맞다.

## ⚠️ 주의·한계
- ISMS-P 기간·비용 수치는 일부 2차 출처를 포함한다. 대외 문서에는 범위값으로만 쓰는 편이 안전하다.
- 원문 로드맵은 법적 통과 경로 중심이다. 실제 구현 난이도와 운영 인력 비용은 별도 산정이 필요하다.
- CSAP은 공공 확장용 축이고, 은행 내부 PoC만 놓고 보면 금융보안원 안전성 평가가 더 직접적이다.
- 샌드박스 지정 사례가 있다고 자동 승인되는 건 아니다. 토큰화 실증과 데이터 흐름 증빙이 선행돼야 한다.
