# D19 요약 — JB LocalGuard OS는 라이선스가 아니라 운영책임을 파는 계약이어야 한다

## ⚡ 30초 핵심
계약가를 올리는 건 모델 토큰비가 아니라 운영통제비다.  
금융권 구매자는 기능보다 로그 보관, DLP, 장애대응, 변경통제, 점검권, 종료 절차를 먼저 본다.  
제안서는 “AI 사용료”가 아니라 “운영 가능한 관리형 서비스” 문장으로 다시 써야 한다.

## 🎯 바로 쓸 것 (에이전트·제품 즉시 활용)
- 가격 문구를 `외부 모델/API + 검색/색인 + 로그·감사보관 + DLP + 백업·DR + 관찰성 + 보안모니터링 + 온콜 + 월간 리포트` 포함형 managed service로 고친다.
- 핵심 SLA 대상을 `케이스 생성`, `위험알림`, `승인대기열`, `감사원장 기록`으로 못 박고, 대고객 자동실행은 계약 범위에서 뺀다.
- 운영 검색 계층은 read-write 기준 최소 `3 replicas` 문구를 넣는다. PoC 가격표를 운영가로 재사용하지 않는다.
- 지원조직을 `서비스 매니저`, `L2 운영`, `L3 AI/LLMOps`, `보안/준법 연락책임자`로 명시하고 P1·보안 사고 `24x7` 연락체계를 넣는다.
- 최고심각도 S0를 `PII 반출 의심`, `감사원장 쓰기 실패`, `승인정책 무단변경`으로 정의한다. 외부 모델 장애·deprecation 통보와 대체모델 절차도 계약 본문에 넣는다.
- 상업적 메시지는 `파일럿 1억 원대 후반~2억 원대`, `정식 운영 3억~5억 원대`, `강화 운영 그 이상` 밴드로 잡되, 공개가격 부재 구간은 산식과 후보 범위로만 제시한다.

## 🙋 전문가에게 묻는다면
- **Q. 왜 라이선스 계약이 아니라 서비스 계약이어야 하나?** A. 금융권은 접속기록 보관, 위탁통제, 재해복구, 점검권, 종료 절차까지 산다. 모델 사용료만 적으면 운영계약처럼 안 보인다.
- **Q. SLA에서 가장 먼저 지켜야 할 대상은?** A. 승인우선 구조인 만큼 고객 자동실행이 아니라 케이스 생성, 알림, 승인 큐, 감사원장이다.
- **Q. 운영비에서 토큰비 비중이 제일 큰가?** A. 아니다. 원문은 로그·감사보관, DLP, 백업·DR, 관찰성, 프리미엄 지원조직이 더 큰 원가를 만든다고 본다.
- **Q. 외부 LLM 장애는 벤더 책임 아닌가?** A. 업스트림 책임과 별개로 고객 통지, 대체모델 전환, 회귀검증, 롤백 절차는 공급사 계약 의무로 둬야 한다.
- **Q. 구매자가 바로 의심하는 실패 신호는?** A. 가용성 숫자만 있는 SLA, 단일 담당자 의존, 로그·DLP 비용 누락, deprecation 절차 부재, 출구전략 없는 계약이다.

## ⚠️ 주의·한계
- Azure AI Search 세부 티어, API Gateway 옵션별 단가, SOC/MDR, 한국형 운영 인건비는 공개 단가가 일관되지 않는다.
- 국내 금융 AI SaaS의 공개 SLA 사례가 적어, 수치·지원등급은 글로벌 클라우드 공식 문서를 많이 빌려왔다.
- 인력비 총액은 FTE-month 산식 중심이다. 제안 직전 내부 loaded rate로 다시 검산해야 한다.
