# D17 요약 — JB금융그룹 데이터 경계와 클린룸 운영 모델

## ⚡ 30초 핵심
같은 그룹이라고 고객 원본 데이터를 자유롭게 섞어 쓰면 안 된다. 기본 구조는 `계열사 내부 실명 레인`, `그룹 공통 토큰 레인`, `전문기관 결합 레인`, `외부 LLM 비식별 요약 레인`의 4단 분리다. PoC는 합성·가명·코드화 데이터부터 시작하고, 식별형 교차판매나 원장 단위 공유는 빼는 게 맞다.

## 🎯 바로 쓸 것 (에이전트·제품 즉시 활용)
- 그룹 공통 Hub에는 원본 고객 원장과 원본 식별자를 올리지 않는다. `subject_token, affiliate_id, domain, risk_code, score_band, evidence_pointer, event_time_bucket, approval_state` 정도의 최소 필드만 둔다.
- 각 계열사 내부에 토큰화 게이트웨이를 두고 `원본키 Vault`, `토큰 생성`, `free-text 마스킹`, `evidence_pointer 생성`을 처리한다. 외부 LLM에는 draft용 비식별 요약만 보낸다.
- PoC 우선순위는 `합성 데이터 -> 공개 데이터 -> 익명/집계 데이터 -> 계열사 내부 가명 데이터 -> 계열사 내부 실명 데이터` 순으로 고정한다.
- 그룹 공통 레이어와 외부 LLM 경로에서 제외할 금지 필드를 명시한다. 주민등록번호, 계좌번호 원문, 카드번호, 휴대폰번호, 정확 주소, 콜 녹취 원문, 상담원 메모 원문, 거래메모 전문, 신분증 이미지, 신용조회사 원본 PDF는 제외한다.
- 계열사 간 행 단위 결합이나 외부 공공데이터 결합이 필요하면 지주 특례 레인과 별도로 `데이터전문기관/결합전문기관 레인`으로 올린다. Case Hub와 클린룸을 섞지 않는다.
- 권한 모델은 `법인 단위 기본 분리 + 목적코드 기반 예외 허용`으로 잡는다. 계열사 간 재식별 조회는 고객정보관리인 승인과 조회이력 기록이 붙어야 한다.

## 🙋 전문가에게 묻는다면
- **Q. 금융지주회사법 특례가 있으면 계열사 간 식별형 고객정보를 바로 써도 되나?** A. 아니다. 내부 경영관리 목적 특례는 있지만 무제한이 아니고, 마케팅·상품권유 업무는 제외된다.
- **Q. 그룹 공통 Hub에 왜 원본 원장을 못 올리나?** A. 감독규정이 고객정보 원장 제공 금지, 승인된 이용자 외 접근 금지, 고유식별정보 암호화·변환을 요구하기 때문이다.
- **Q. 업무위탁과 공동이용은 어떻게 다르나?** A. 위탁은 수탁자가 자기 목적 사용을 못 하고, 공동이용·제3자 제공은 수령자가 자기 목적을 가진다. 설계와 동의 구조가 달라진다.
- **Q. 가명처리와 클린룸은 같은 말인가?** A. 아니다. 가명처리는 데이터 처리 방식이고, 클린룸은 결합·분석·반출을 통제하는 운영 공간이다.
- **Q. 외부 LLM은 어디까지 써도 되나?** A. 비식별 요약 기반의 초안 작성 보조까지다. 고객 대상 행동과 최종 판단은 내부 승인 게이트 뒤에 남겨야 한다.

## ⚠️ 주의·한계
- JB금융그룹 내부 실제 승인 워크플로, 위탁계약 표준조항, 데이터 카탈로그, 외부 LLM 계약 조건은 공개자료만으로 확인되지 않았다.
- 지주 특례 레인으로 처리할 시나리오와 전문기관 결합 레인으로 올릴 시나리오는 내부 정책 문서로 먼저 확정해야 한다.
- 개인사업자·법인·대표자보증이 섞인 필드는 개인신용정보성 분류가 상품 구조마다 달라질 수 있다.
- 최소 샘플 수, 셀 억제 기준, 재식별 위험 허용치, 프롬프트 차단 룰 같은 정량 반출 기준은 JB 내부에서 별도 제정이 필요하다.
