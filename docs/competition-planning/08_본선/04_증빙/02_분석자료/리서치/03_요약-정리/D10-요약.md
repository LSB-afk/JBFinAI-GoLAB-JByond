# D10 요약 — 외부 프런티어 LLM을 은행 망분리 안에서 쓰는 설계 원칙

## ⚡ 30초 핵심
외부 LLM을 쓰더라도 고객 원본 PII와 신용정보는 밖으로 내보내지 않는 구조가 전제다. 해법은 모델 신뢰가 아니라 브로커 통제다. 토큰화 게이트웨이, 단일 egress 프록시, 도구 최소권한, 사람 승인, tamper-evident 감사로그를 한 세트로 붙여야 한다.

## 🎯 바로 쓸 것 (에이전트·제품 즉시 활용)
- 외부 호출 앞단에 `데이터 분류 -> PII 탐지 -> 토큰화/가명화 -> 최소 맥락화 -> 외부 호출 -> 응답 DLP 스캔` 순서의 브로커를 둔다.
- 주민번호·계좌번호·전화번호는 `FPE` 또는 deterministic tokenization, 고객명·주소·자유서술은 룩업형 불투명 토큰으로 나눈다.
- 외부 LLM은 "권한 없는 제안 엔진"으로만 둔다. 송금, 변경, 통지, 다운로드, 색인 갱신은 툴 브로커와 정책 엔진이 막고 고위험 작업은 이중승인으로 보낸다.
- 툴은 서비스 계정 분리, 최소 scope, allow-list 파라미터, rate limit, `kill switch`를 기본값으로 둔다.
- 출력은 그대로 실행하지 않는다. HTML, Markdown, SQL, path, command, tool parameter는 스키마 검증과 context-aware encoding 뒤에만 downstream으로 넘긴다.
- 레드팀 테스트셋에 직접·간접 prompt injection, hidden instruction, tool call injection, PII echo, SQL/HTML payload, excessive agency 시나리오를 넣고 배포 전후 반복 측정한다.

## 🙋 전문가에게 묻는다면
- **Q. 외부 LLM을 써도 되는 최소 조건은?** A. 원본 비반출이다. 외부 모델에는 토큰화된 일부 컨텍스트와 추론 질문만 보낸다.
- **Q. 프롬프트 인젝션은 막을 수 있나?** A. 0으로 못 막는다. OWASP도 fool-proof 방법이 불분명하다고 적는다. 그래서 권한과 반출량을 줄이는 쪽이 맞다.
- **Q. 제일 위험한 실패 지점은 어디인가?** A. 출력 처리와 과도한 권한이다. 모델 답변이 그대로 실행되거나 조회용 에이전트가 변경 권한까지 가지는 순간 사고로 이어진다.
- **Q. 공급자 보안 옵션만 믿으면 되나?** A. 아니다. `30일 로그 보관`, 서비스 처리면, 로그·캐시 잔존, 재식별, 도구 남용 문제는 브로커 밖에서 계속 남는다.
- **Q. 감사로그는 무엇까지 남겨야 하나?** A. 사용자·서비스 ID, 토큰화된 요청 요약, 정책 결정 이유, 모델/버전, 툴 호출, 승인 여부, DLP 결과, 이전 로그 해시 또는 서명까지 남긴다.

## ⚠️ 주의·한계
- 국내 은행의 실제 내부 LLM 보안 토폴로지는 공개 자료가 적다. 원문은 공개 표준과 공개 사례를 묶어 권고 아키텍처를 제시한다.
- 한국어 비정형 금융문서에서 PII 탐지 정밀도 기준은 공개 1차 자료가 제한적이다.
- 토큰화만으로 끝나지 않는다. 준식별자 조합과 비정형 탐지 누락이 남는다.
- 단방향 데이터 다이오드는 실시간 요청-응답 챗봇에 단독으로 맞지 않는다.
