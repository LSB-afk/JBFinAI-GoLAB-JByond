# D22 요약 — 금융권 AI 모델 도입 전략과 로컬 우선 인프라 경제성

## ⚡ 30초 핵심
은행의 외부 LLM 거부감은 보수성이 아니라 고객정보 반출, 제3자 IT 리스크, 감사 책임, 모델·데이터 검증 책임 때문이다.
정답은 “외부 금지”가 아니라 로컬 기본값, 정책 게이트, 비식별 승격, 감사 로그를 갖춘 하이브리드다.
비용도 전기료가 아니라 GPU 활용률과 모델 크기가 가른다. 대량·반복·민감 업무는 로컬, 희소·고난도·공개정보 업무는 premium API가 맞다.

## 🎯 바로 쓸 것 (에이전트·제품 즉시 활용)
- 기본 경로는 `PII 비반출`: 고객식별정보·상담기록·내부 규정·상품문서는 로컬 모델/RAG만 통과. 외부 호출은 정책엔진이 차단하거나 승인 대기시킨다.
- 모델 라우팅은 4층으로 잡는다: `8B~32B 로컬 내재화` → `HyperCLOVA X SEED/Solar Open 국산 우선 평가` → `Llama 3.1/Qwen3/DeepSeek distill 오픈웨이트` → `외부 premium API 예외 승격`.
- 승격 기준은 난이도가 아니라 데이터 등급이다: 공개정보, 비식별 요약, 합성 테스트셋만 외부로 보낸다. 민감정보가 섞이면 로컬로 강등한다.
- 하네스는 model-agnostic으로 설계한다. LangGraph/Deep Agents식으로 모델·도구·상태·오케스트레이션을 분리하고, human-in-the-loop·평가·추적을 붙인다.
- 시연 숫자는 그대로 쓴다: H100 1장 Llama 3.1 8B 214.56 tok/s, H100 4장 Llama 3.1 70B 387.55 tok/s, H200 2장 Llama 3.3 70B 261.83 tok/s.
- 비용 메시지는 선명하게 간다: 4xH100 70B는 GPT-4.1 mini보다 싸지려면 월 약 97억 토큰이 필요하지만, GPT-5.5 대비 break-even은 70/30 기준 약 5.9억 토큰이다.

## 🙋 전문가에게 묻는다면
- **Q. 금융권은 왜 외부 LLM API를 꺼리나?** A. 학습 여부보다 외부 전송 자체가 문제다. OpenAI API도 기본 무학습이지만 남용 모니터링 로그를 최대 30일 보관할 수 있다.
- **Q. 그럼 완전 온프레만 답인가?** A. 아니다. 온프레, 전용·지역 제한 클라우드, 게이트웨이·프록시, 데이터 비전송 설계를 업무별로 섞는 구조가 현실적이다.
- **Q. 중국계 모델은 무조건 배제해야 하나?** A. 아니다. DeepSeek 앱/서비스 리스크와 셀프호스팅 오픈웨이트 리스크는 다르다. 다만 금융권 대외 설명 부담은 크다.
- **Q. 작은 로컬 모델로 실무가 되나?** A. 단독 채팅은 약하다. 대신 function calling, 내부 API, SQL/룰엔진, RAG, 검증 루프를 붙이면 규정 QA·분류·요약·템플릿 업무는 충분히 설득력 있다.
- **Q. 로컬 LLM TCO에서 전기료가 큰가?** A. 아니다. 예시상 4xH100급 월 전력·기본요금은 약 26.6만원 수준이다. 더 큰 변수는 GPU 감가상각, 운영인력, 활용률이다.

- 모델 가격·라이선스·성능 비교는 2026-07-01 접속 기준이다. 제출 전 최신 가격표와 라이선스를 다시 확인해야 한다.
