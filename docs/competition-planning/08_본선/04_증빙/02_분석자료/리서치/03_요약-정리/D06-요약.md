# D6 요약 — 은행 원장은 건드리지 말고 CDC·Nexacro·MCI로 우회 연결한다

## ⚡ 30초 핵심
한국 은행 시스템은 `계정계`, `정보계`, `채널계`가 분리된 구조라 AI 콘솔도 그 경계에 맞춰 붙어야 한다.  
읽기는 `계정계 로그 → CDC → 내부 가명처리 → Kafka → AI 콘솔`이 맞고, 쓰기는 AI가 직접 하지 말고 RM 단말의 승인 후 `MCI/EAI`를 타야 한다.  
전북은행과 광주은행은 공동 전산센터 계획을 철회했으니, 연동도 두 은행을 별도 호스트·별도 프로젝트로 봐야 한다.

## 🎯 바로 쓸 것 (에이전트·제품 즉시 활용)
- 읽기 경로는 `Debezium 같은 로그 기반 CDC`로 고정한다. 운영 DB 직접 조회나 계정계 API 추가는 피한다.
- PII 경계는 `은행 내부 Tokenization Proxy`에 둔다. AI 콘솔은 이름·계좌번호 대신 해싱된 토큰 ID와 위험 피처만 받는다.
- RM 화면은 `Nexacro WebView/WebBrowser`에 LocalGuard 웹앱을 임베딩하고 `postMessage`, `callMethod`, `onusernotify` 브릿지로 붙인다.
- 쓰기 경로는 `AI 권고 → RM 승인 클릭 → Nexacro 런타임 → MCI/EAI 표준 전문 → 계정계/발송계` 순서로 설계한다.
- 외부 공개 API는 `오픈뱅킹`, `마이데이터` 규격 참고용으로만 보고, 내부 연동은 `MCI`, `EAI`, `API Gateway` 어댑터를 따로 설계한다.
- 전북은행과 광주은행은 통합전산센터가 아니라 개별 전산센터 방향이므로 배포·연동·보안심사를 이원화한다.

## 🙋 전문가에게 묻는다면
- **Q. 왜 AI 콘솔이 계정계 DB를 직접 읽으면 안 되나?** A. OLTP 부하와 원장 가용성 때문이다. 원문은 로그 기반 CDC를 가장 현실적인 Read-only 경로로 본다.
- **Q. 왜 쓰기도 AI가 바로 하지 않나?** A. 내부 통제상 외부 솔루션의 직접 Write는 막혀 있다. 승인 뒤 단말 런타임이 기존 MCI 채널을 호출해야 한다.
- **Q. RM 단말에는 어떻게 화면을 넣나?** A. 넥사크로 WebView 임베딩 후 가상 토큰 UUID만 주고받는 방식이다.
- **Q. 외부 오픈뱅킹 API를 내부 연동에 그대로 쓰면 되나?** A. 아니다. 그 규격은 대외 표준일 뿐이고 내부망은 MCI/EAI의 비공개 전문 규격을 맞춰야 한다.
- **Q. JB 두 은행을 한 번에 묶어 붙일 수 있나?** A. 어렵다. 2025년 통합 전산센터 계획이 철회돼 각자 독립 환경으로 보는 게 맞다.

## ⚠️ 주의·한계
- 전북은행·광주은행 단말의 실제 WebView 엔진 버전과 최신 프런트 기능 호환성은 미검증이다.
- MCI/EAI 표준 전문 스키마와 암호화 규칙은 비공개라 초기 보안 연계 검토 없이는 확정할 수 없다.
- 개별 IDC 신설 이후의 물리 라우팅과 Active/Backup 구조는 2029년 가동 전까지 외부에서 알기 어렵다.
