# D15 요약 — 승인우선·감사원장 설계가 실제로 막는 것과 못 막는 것

## ⚡ 30초 핵심
`승인 먼저, 감사원장 기록` 방향은 금융 규제와 잘 맞는다. 하지만 사람 승인만 붙인다고 안전해지진 않는다. 진짜 차이는 rubber-stamping 방지, false block 복구, kill switch, immutable evidence, incident reporting까지 구조로 강제하느냐에서 난다.

## 🎯 바로 쓸 것 (에이전트·제품 즉시 활용)
- 승인 게이트는 `Approve / Reject` 버튼으로 끝내지 않는다. 모델 출력, 입력 스냅샷, 규칙·프롬프트·모델 버전, 데이터 출처, 정책 충돌 여부를 한 화면에 같이 보여준다.
- 승인자에게 이유 코드와 자유서술 근거를 강제한다. 고위험 행위는 이중승인, 직무 분리 승인, 반대·재심·상향보고 경로를 같은 강도로 제공한다.
- false block 대비로 `차단 범위 최소화`, `부분 제한 우선`, `5영업일 내 결과 통보` 수준의 SLA, 임시 해제 권한, 복구 플로우를 넣는다.
- kill switch는 UI 버튼이 아니라 실제 채널·배치·서드파티 API·메시징 큐까지 끊어야 한다. 발동 즉시 사고 티켓, 영향 고객 집계, 감독보고 초안, fallback 모드를 자동 생성한다.
- 감사 원장은 append-only 저장, 시간 동기화, 해시/전자서명, 변경 히스토리, 관리자 접근기록, forensic export 패키지를 기본으로 둔다.
- KPI는 승인 건수보다 `override 비율·사유`, `false block 해제 SLA`, `설명요구 처리기한 준수율`, `kill switch 발동 시간`, `감사 샘플 적합률`로 잡는다.

## 🙋 전문가에게 묻는다면
- **Q. 승인우선 설계의 제일 큰 장점은?** A. 무단 실행 억제, 책임 추적, 감독 대응용 증거 확보, 설명권 대응이다.
- **Q. 그럼 왜 이것만으로 부족한가?** A. rubber-stamping, 입력 데이터 오염, 벤더 유출, 경보 무시, kill switch 미연결, 승인 대기 병목은 그대로 남기 때문이다.
- **Q. 실사례가 있나?** A. Knight Capital은 45분 동안 400만 건 넘는 오류 주문으로 4억6천만 달러 초과 손실을 냈다. Wells Fargo는 faulty automated filter로 100만 개 넘는 계좌를 동결했다.
- **Q. 감사 원장은 어떤 수준이어야 법적 증거가 되나?** A. 로그가 있다는 것만으론 부족하다. 무결성, 동일성, chain-of-custody를 입증할 수 있어야 한다.
- **Q. 국내 규제와도 맞나?** A. 맞다. 내부통제기준, 접속기록 보존, 자동화평가 설명·이의제기, 사고 최초보고 24시간 이내 체계와 직접 연결된다.

## ⚠️ 주의·한계
- 국내 금융회사에서 approval-first console이 사고를 몇 퍼센트 줄였는지 보여주는 공개 1차 자료는 부족하다.
- `automation bias`라는 용어를 국내 판례·제재문에 직접 붙인 자료는 확인되지 않았다.
- 블록체인 자체가 필수라는 규제 요구는 없다. 기술보다 무결성 요건 충족이 먼저다.
- 감사 원장은 사후 책임 추적에는 강하지만, 개발·학습 단계 데이터 경계가 무너지면 선제 예방엔 한계가 있다.
