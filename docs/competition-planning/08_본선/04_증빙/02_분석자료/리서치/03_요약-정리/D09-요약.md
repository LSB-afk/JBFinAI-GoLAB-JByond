# D9 요약 — 금융형 에이전트 콘솔은 RAG, 규칙, 승인게이트, 추적을 분리해서 쌓는다

## ⚡ 30초 핵심
외부 레퍼런스가 수렴하는 기본 구조는 `인입·정규화 → 청킹·임베딩·인덱싱 → 검색 → LLM 추론·오케스트레이션 → 후처리·검증 → 도구 실행 → 로깅·추적`이다.  
금융권 차별점은 새 레이어보다 `토큰화·마스킹`, `정책 게이트`, `승인 hold`, `버전·도구·세션 추적`을 더 강하게 넣는 데 있다.  
MVP는 `classic RAG`로 시작하고, 이후 `hybrid/rerank/citation`, `tool schema`, `approval gate`, `multi-agent`, `rollout router` 순서로 키우는 게 맞다.

## 🎯 바로 쓸 것 (에이전트·제품 즉시 활용)
- 검색 스택은 `classic RAG + hybrid retrieval + reranking + citations`를 기본값으로 둔다.
- 규정·약관·심사자료는 pure vector만 쓰지 말고 키워드 검색을 같이 둔다. 조항 번호, 날짜, 상품 코드는 exact match가 더 강하다.
- PDF·이미지·표 문서는 OCR·document extraction 품질이 검색 품질을 결정하므로 전처리 계층을 따로 둔다.
- 결정형 결과는 `BRMS/DMN/정책 엔진`이 최종 게이트를 맡고, LLM은 요약·추출·설명·추천 계층으로 제한한다.
- 액션 호출은 정적 함수가 아니라 `OpenAPI action group` 또는 `MCP/tool schema`로 승격하고, 승인 전에는 `interrupt/checkpoint/tool approval` 상태로 hold한다.
- 운영은 `prompt/model/agent 버전 분리`, `OpenTelemetry/OpenInference 기반 trace`, `API gateway/custom router`, `independent rollback` 구조로 짠다.

## 🙋 전문가에게 묻는다면
- **Q. 금융형 RAG의 기본 패턴은 뭔가?** A. classic RAG에 hybrid search, reranking, citations를 붙이는 게 가장 안정적이다.
- **Q. 규칙 엔진과 LLM 중 누가 최종 결정을 내리나?** A. 최종 승인·차단·적격성 판정은 규칙 계층이 맡고, 모델은 전후에서 보조한다.
- **Q. 멀티에이전트가 꼭 처음부터 필요한가?** A. 아니다. 단일 에이전트나 classic RAG로 시작하고 필요할 때 specialist와 handoff를 추가하는 경로가 일반적이다.
- **Q. 승인 게이트는 기술적으로 구현 가능한가?** A. 가능하다. LangGraph `interrupt`, Microsoft Agent Framework `tool approval`, Bedrock `user confirmation`이 그 패턴이다.
- **Q. 내부 모델과 외부 프런티어 모델을 섞는 게 이상한가?** A. 아니다. self-hosted model과 managed model 혼용은 금융·엔터프라이즈 문서에서 반복되는 표준 선택지다.

## ⚠️ 주의·한계
- 은행이 직접 공개한 상세 아키텍처 다이어그램은 희소해서, 세부 구현 일반화에는 한계가 있다.
- Google revisions/traffic splitting, 일부 agentic retrieval 기능 등은 2026 시점 preview·beta 성격이 섞여 있다.
- 이 문서는 구현 아키텍처 패턴 중심이다. 보안 위협, 프롬프트 인젝션, KPI 설계는 별도 트랙으로 봐야 한다.
