# D11 요약 — 케이스 메모리형 에이전트 콘솔을 설계하는 학술 기준

## ⚡ 30초 핵심
메모리는 한 덩어리 저장소가 아니다. 작업기억, 일화기억, 의미기억, 절차기억을 분리해야 재사용성과 거버넌스가 같이 선다. 케이스 단위 콘솔은 `원본 이벤트 + 구조화 요약 + provenance + 승인/감사`를 묶는 쪽이 맞다.

## 🎯 바로 쓸 것 (에이전트·제품 즉시 활용)
- `Case -> AgentRun -> Skill -> Evidence -> Approval -> Audit` 구조를 유지하되, AgentRun은 일화기억, Skill은 절차기억, Evidence는 provenance가 붙은 의미기억으로 정의한다.
- 케이스 최소 스키마는 `문제 상황, 제안된 해결, 실행 결과, 설명/교훈`으로 둔다. 최종 답만 저장하지 말고 왜 그렇게 판단했는지와 결과까지 남긴다.
- 메모리 라이프사이클을 `write -> retrieve -> update -> consolidation/reflection -> retention/forgetting`으로 설계한다. 실행 로그 원본과 정제 요약은 분리 저장한다.
- 검색은 문서 조각 RAG보다 `케이스 RAG` 우선으로 잡는다. 유사 케이스 검색 결과를 그대로 넣지 말고 revise 단계와 인간 검토를 붙인다.
- provenance 필드를 1급으로 둔다. 출처, 생성자, 수정자, 시각, 버전, 변경 이력이 없으면 장기지식이 아니라 비감사성 캐시로 떨어진다.
- 실패 케이스와 반례도 같이 저장한다. 성공사례만 쌓이면 case-base 편향과 확증편향이 커진다.

## 🙋 전문가에게 묻는다면
- **Q. 왜 메모리를 네 종류로 나누나?** A. 현재 작업 상태, 사건 이력, 일반화된 규칙, 실행 절차는 쓰임이 다르다. 한 저장소에 섞으면 검색도 설명도 망가진다.
- **Q. 케이스 중심이 문서 중심 RAG보다 나은 점은?** A. 사례 단위로 문제, 해결, 결과, 교훈을 묶어 재사용한다. 그래서 설명과 적응이 쉽다.
- **Q. 승인과 감사가 왜 메모리 모델 안에 들어가나?** A. CBR의 `Revise`와 `Retain`을 제도화한 것이 Approval과 Audit이기 때문이다.
- **Q. 가장 위험한 실패모드는?** A. 메모리 오염, 잘못된 유사도, 성공사례 편향, 프라이버시 누적, 오래된 요약의 모순 누적이다.
- **Q. 한국 규제와도 맞물리나?** A. 맞다. 자동화된 결정 설명 요구와 신용정보 자동화평가 설명 체계는 근거, 기준, 입력정보, 판단시점 저장을 요구하는 방향이다.

## ⚠️ 주의·한계
- 금융 규제 실무와 LLM 케이스 메모리를 정면으로 연결한 1차 실증연구는 아직 적다.
- 2025~2026 메모리 서베이 일부는 arXiv·OpenReview 단계라 보조 근거로만 써야 한다.
- `HagentOS` 비교는 원문도 일반 원리 수준으로만 다룬다. 특정 제품 정의 검증은 하지 않았다.
- 장기 메모리는 삭제·정정·재산출 인터페이스 없으면 프라이버시 리스크가 빠르게 커진다.
