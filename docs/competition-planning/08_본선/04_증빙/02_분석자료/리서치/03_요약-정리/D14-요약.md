# D14 요약 — RM용 거버넌드 AI 콘솔의 데이터·모델 신뢰성 점검

## ⚡ 30초 핵심
이 제품은 자동판정 엔진보다 승인형 보조도구로 설계하는 편이 법과 데이터 현실에 맞다. 문제는 모델 성능 하나가 아니라 thin-file, 전세 데이터 오류, FDS 표류, 한국어 금융 모델 공백이 동시에 있다는 점이다. 해법은 `인간 승인 + 데이터 신뢰도 게이트 + 설명패킷 + 재심 파이프라인`이다.

## 🎯 바로 쓸 것 (에이전트·제품 즉시 활용)
- 각 판단마다 `모델 버전, 사용 데이터셋 시점, 출처별 freshness, 주요 사유코드, 원문 근거 스냅샷, 승인자, override 사유`를 묶은 결정 패키지를 저장한다.
- 이의제기는 민원처리가 아니라 재산정 파이프라인으로 설계한다. 데이터 정정 후 같은 규칙·같은 모델로 재평가하고 최초 결과와 차이를 로그로 남긴다.
- 공정성은 단일 점수로 끝내지 않는다. 승인율 격차, TPR 격차, 오류율 격차, 사유코드 분포, 인간 override 편차를 모델 수준과 운영 수준으로 나눠 본다.
- 소상공인·전세 입력에는 `생성시점, 수집시점, 마지막 갱신시점, 결측률, 이상치, 원천-정합성`을 계산하고 데이터 신뢰도 점수가 낮으면 자동 판단을 막고 수기검토로 보낸다.
- 사기 탐지는 룰+모델 혼합으로 간다. 새 패턴은 룰로 즉시 막고, 모델은 drift·라벨 지연·precision/recall을 배치 검증한 뒤 승격한다.
- 생성 모델 하나에 맡기지 말고 `한국어 금융 임베딩/분류기 + 내부 문서 RAG + 규칙엔진 + 인간 승인` 조합으로 간다. 외부 LLM은 설명 초안과 추가 확인 질문 생성 정도로 좁힌다.

## 🙋 전문가에게 묻는다면
- **Q. 왜 완전자동 승인/거절이 아니라 승인형 보조도구인가?** A. 신용정보법과 개인정보보호법이 설명, 거부, 이의제기, 재검토 권리를 준다. 중대한 불이익 판단은 인간 재검토 경로가 필요하다.
- **Q. 공정성은 어느 정도로 보나?** A. 한국에 safe harbor 숫자는 없다. 그래서 승인율, TPR, 오류율, 사유코드, override 편차를 같이 보는 조기경보 체계가 현실적이다.
- **Q. 데이터 품질에서 제일 약한 구간은?** A. 소상공인 thin-file, 전세 시세·등기·실거래의 시차와 오류, FDS의 class imbalance와 concept drift다.
- **Q. 한국어 금융 모델만으로 충분한가?** A. 아직 아니다. 원문 기준 공개모델은 지식 점수는 올라가도 추론과 안전성 격차가 남는다.
- **Q. 공개 데이터로 파인튜닝하면 끝나나?** A. 아니다. 공개 데이터는 작거나 합성·제한 공개가 많다. 실무 커버리지는 별도 데이터 확보가 필요하다.

## ⚠️ 주의·한계
- 한국 금융회사들이 실제 쓰는 공정성 임계치와 집단별 성능치는 거의 공개되지 않았다.
- 전세사기용 공개 대규모 라벨 데이터셋은 이번 조사에서도 강하게 확인되지 않았다.
- 공개 한국어 금융 데이터는 많아 보여도 안심존, IRB, 반출통제 제약이 큰 편이다.
- 금융위 안내서 일부는 존재와 방향은 확인됐지만 전문 본문까지 직접 판독한 1차 인용은 제한적이었다.
