# D25 요약 — 데이터 레인은 원본 PII 비반출, 승인 우선, 계열사 최소공유로 자른다

## ⚡ 30초 핵심
외부 LLM에 원본 PII·개인신용정보를 보내지 않는 구조가 가장 방어적이다.  
고객 접점 실행은 사람 승인 뒤에만 열고, 계열사 이식은 식별형 원장 통합이 아니라 최소정보·단기이용·분리보관으로 설계해야 한다.  
핵심은 기술 미학이 아니라 `레인 분리`다.

## 🎯 바로 쓸 것 (에이전트·제품 즉시 활용)
- 데이터 레인을 `공개형·집계형 → 외부 LLM`, `개인신용정보·원본 PII → 내부 레인`, `내부 생성물 → Zero-PII 파생 레인`으로 분리한다.
- 외부 LLM에는 `토큰`, `위험코드`, `근거 포인터`, `승인상태`, `모델 버전`만 보낸다. 이름, 주민번호, 연락처, 계좌번호, 신용점수 원문, 상담 원문은 원천 시스템에 남긴다.
- 고객 메모리는 원장 통합이 아니라 Zero-PII 메모리로 설계한다. 저장 대상은 `조치 이력`, `승인 이력`, `설명 생성 근거`까지로 제한한다.
- 반출 입증은 `반출 프록시 로그 + DLP 판정 로그 + 토큰화 로그 + 승인 로그 + 모델/리전/보존옵션 로그` 묶음으로 남긴다.
- 계열사 구조는 `고객 응대·실행은 각 계열사 식별형`, `그룹 모델·리스크 분석은 가명형 결합`, `대외 설명·외부 LLM 연결은 익명·Zero-PII 파생물` 3단으로 고정한다.
- 보존정책은 `개인신용정보 수집·이용·제공·폐기 기록 3년`, `개인정보처리시스템 접속기록 1년 또는 2년`, `전자금융거래기록·중요원장 직접기록 5년`을 기준으로 깔고, RM 초안·중간 프롬프트는 단기 파기 원칙으로 둔다.

## 🙋 전문가에게 묻는다면
- **Q. 토큰화하면 외부 LLM 반출이 자유로워지나?** A. 아니다. 원본보다 훨씬 낫지만 대체로 가명정보다. 복원 키가 내부에 있으면 무규제라고 말할 수 없다.
- **Q. 계열사끼리 식별형 고객정보를 바로 합쳐도 되나?** A. 상시 원장 통합은 위험하다. 금융지주 특칙은 가능성을 열지만 목적, 이용자, 기간, 분리보관, 승인 절차를 강하게 건다.
- **Q. 외부 생성형 AI 호출은 이미 정규 허용 상태인가?** A. 아니다. 원문 기준 2026-07-02 시점에는 SaaS 예외는 시행됐지만 생성형 AI는 여전히 샌드박스 활용·연내 제도화 추진 단계다.
- **Q. Zero-PII 메모리가 왜 유리한가?** A. 최소수집, 목적구속, 파기, 설명가능성을 동시에 맞추기 쉽다. 원본 고객정보 영속 메모리보다 방어가 낫다.
- **Q. 공개형 데이터는 그냥 외부로 보내도 되나?** A. 아니다. 라이선스뿐 아니라 자연인 식별요소가 섞였는지 다시 봐야 한다.

## ⚠️ 주의·한계
- `Zero-PII`는 법적 무규제 구역이 아니다. 재식별 가능성이 있으면 여전히 개인정보 또는 가명정보 평가를 받는다.
- 상용·대체데이터는 법보다 계약이 더 세게 작동한다. 계열사 공유, 재학습, 재배포 금지 조항을 별도 검토해야 한다.
- 국외이전 쟁점이 남는다. 외부 LLM은 리전, 서브프로세서, 원격지원, 로그 저장 위치까지 계약에 적어야 한다.
