# B1 — 롤모델 오픈소스 벤치마크 · 소스 리프트 가능성

> **B-시리즈 = 빌드 레퍼런스 트랙** (제품을 *실제로 만들기 위한* 외부 코드·디자인 본보기 정찰).
> D-시리즈(도메인 지식·근거·인용)와 목적이 다름 — 여기선 "무엇을 베껴 와서 며칠 안에 작동시킬 수 있나"가 핵심.

> ✅ **Phase 1 회수 완료(2026-07-01, ChatGPT Deep Research)** → [[B01-결과-원문-gpt]]. 결론: 기능=**StackStorm**+**Temporal**, 디자인=**Carbon**+**Tremor**. AGPL/GPL 계열(TheHive·Windmill·n8n·Midday·Twenty·Grafana)은 클린룸 재구현만. ⏳ Phase 2(로컬 티어다운) 대기.

## 메타 (프롬프트 밖 — 생성·운영용)
- **도메인**: JB LocalGuard OS의 **기능·디자인 롤모델이 될 오픈소스 프로그램** 정찰 + **소스/디자인 리프트(차용) 가능성·이식 난이도** 평가.
- **목적**: 사용자 우려 직격 — "정말 그 오픈소스급으로 작동해야 하는데 며칠 안에 되나?" → 막연한 "멋진 앱 구경"이 아니라, **(a) 라이선스상 합법 차용 가능 + (b) 우리 스택으로 며칠 내 리프트 가능한 구체적 조각**을 골라내는 것.
- **심사 연결**: 3(AI Agent 설계·기술 구현 가능성 — "현실적으로 구현 가능" 방어) · 4(MVP 완성도) · 5(디자인·완성도, 신뢰감).
- **2단계 설계**:
  - **Phase 1 (지금 · 웹 정찰)**: 후보 오픈소스 숏리스트 + 점수화. 외부 딥리서치 엔진(GPT/Gemini)이나 research 에이전트로 실행.
  - **Phase 2 (소스 폴더·스크린샷 도착 후 · 로컬 티어다운)**: 사용자가 실제 OSS 폴더와 캡처/이미지를 가져오면 builder·designer 에이전트가 코드·디자인을 분해해 **구체 리프트 플랜(파일 단위·일수 추정)** 산출.
- **경계 메모**:
  - **B1 ↔ D9**: D9 = 업계 *추상 레퍼런스 아키텍처·패턴*(우리가 안 베낌). B1 = *구체적 실제 레포*를 **베껴 올 대상으로** 평가(라이선스·이식난이도까지). 침범 아니라 보완 — D9 결과를 인용해 "이 패턴을 구현한 실제 OSS가 이것"으로 잇는다.
  - **B1 ↔ designer(paperclip 분석)**: paperclip = 우리 1차 디자인 본보기(이미 분석됨). B1의 디자인 축은 **추가 본보기·OSS 디자인시스템**을 넓혀 paperclip을 보강.
  - **스택 현실 고정**: 우리 MVP = **바닐라 JS(프레임워크·번들러 없음), 문자열 템플릿 렌더, localStorage**. 본선 목표 = 핵심 함수계약(`computeRiskDecision`·`buildDashboardData`·`auditChainRecords`·`moveCaseToColumn`)을 서버 API로 승격. → 대부분의 폴리시드 OSS는 React/TS이므로 **"런타임 코드 통째 리프트"는 거의 불가**, **디자인 토큰·CSS·레이아웃·IA·알고리즘·스키마 리프트가 현실적**이라는 점을 모든 평가에 강제.
  - **하드룰**: 외부 LLM에 원본 PII 비반출 · 고객대상 액션은 사람 승인 게이트 — 이를 위반하는 구조는 본보기에서 제외.
- **제품 연결**: 운영계약 `Case → AgentRun → Agent → Skill → Evidence → Approval → Audit`. 이 7요소를 실제로 구현한 OSS를 찾아 **승인 게이트·감사체인·에이전트 오케스트레이션**의 작동 코드와 신뢰감 있는 콘솔 UI를 본보기로 확보.

## Phase 1 — 웹 정찰 딥리서치 프롬프트 (이 아래 통째로 붙여넣기)

당신은 **엔터프라이즈/금융 운영 콘솔과 AI 에이전트 오케스트레이션 제품의 오픈소스 생태계를 정찰하는 기술·디자인 벤치마크 리서처**다. 목표는 아래 제품의 **롤모델이 될 오픈소스 프로그램**을 (a) 기능 면과 (b) 디자인 면에서 찾아내고, 각 후보에 대해 **우리가 실제로 무엇을 합법적으로 차용(lift)할 수 있고, 우리 스택으로 옮기는 데 며칠 수준의 노력이 드는지**를 냉정하게 평가하는 것이다.

**평가 대상 제품(JB LocalGuard OS)**
- 정체: 지역은행 RM/심사역을 위한 **금융 AI-에이전트 운영 콘솔**. 고객 리스크 신호를 하나의 `Case`로 모아, 전문 AI 에이전트(판단 → 행동 초안 → 검증)를 돌리고, **모든 고객대상 액션을 사람 승인 게이트 뒤에 둠**. **원본 고객 PII는 외부 LLM에 절대 반출하지 않음.**
- 운영계약: `Case → AgentRun → Agent → Skill → Evidence → Approval → Audit`.
- 현재 스택(엄수 — 이식성 판단의 기준): **바닐라 JavaScript(프레임워크·번들러 없음), 문자열 템플릿 HTML 렌더, localStorage 상태.** 본선 목표는 핵심 함수계약을 서버 API로 승격.
- 한국 금융권 맥락: 망분리·금융클라우드, 온프레미스/프라이빗 배치 친화.

**스코프·관점(엄수)**: 단순 "유명한 멋진 앱 나열"이 아니다. 각 후보를 **"우리 롤모델로 합법적·현실적으로 베껴 올 수 있는가"** 관점에서 평가하라. 라이선스(차용 가능 여부)와 우리 바닐라-JS 스택과의 거리를 **모든 후보에 반드시** 매긴다. "이건 멋지다"가 아니라 "이것의 X를 며칠 안에 우리 것으로 가져올 수 있다 / 없다".

### 알아내야 할 것 (핵심 질문)

**A. 운영계약 정합 — 기능 롤모델(케이스·승인·감사)**
1. 우리의 `Case → Agent → Evidence → Approval → Audit` 루프와 **구조적으로 가장 닮은 오픈소스**: SOAR/보안사고대응 케이스관리(TheHive+Cortex, Shuffle, StackStorm, Catalyst 등), 인시던트 운영, 티켓/케이스 시스템. 각각이 우리 7요소 중 무엇을(케이스·에이전트실행·증거·승인·감사) **기본 제공**하는지 매핑하라.
2. 그중 **human-in-the-loop 승인 게이트**와 **변조방지 감사추적(audit trail)**을 *코드로 실제 구현*한 것은 무엇인가. 그 구현이 어디에(어떤 모듈/패턴) 있는가.

**B. AI 에이전트 오케스트레이션 + HITL**
3. 에이전트 오케스트레이션 OSS(Dify, Langflow, Flowise, n8n, Windmill, Inngest, Trigger.dev, Temporal 등) 중 **오케스트레이터-워커 / 계획-실행-검증 / 승인 게이트 / 단계별 증거·관측(observability) / 도구권한 제한**을 노출하는 것. 스택·라이선스·성숙도·기본 UI 품질을 정리.

**C. 승인 워크플로 · 감사증거 빌딩블록**
4. 승인 큐·감사체인을 **부품으로 재사용**할 수 있는 OSS: 워크플로 엔진(Camunda/Zeebe·Temporal·Windmill), 감사로그 라이브러리, 인가(OpenFGA·Casbin). 우리가 서버 API 단계에서 끼워 넣을 만한 합법·경량 부품 후보.

**D. 디자인 롤모델 — 신뢰감 있고 따라할 수 있는**
5. **데이터 밀도 높은 운영 콘솔 UI**의 본보기가 될 OSS 앱(예: Twenty, Plane, Cal.com, Dify, OpenBB, Grafana 등 — 실제 후보는 네가 검증)과, **문서화된 엔터프라이즈 디자인시스템**(Carbon/IBM, Atlassian, Shopify Polaris, Untitled UI, Tremor, Radix, Geist 등). 각각에서 **따라할 수 있는 구체 패턴**(컬러·간격·라운드·타이포 토큰, 그리드/레이아웃, 정보구조(IA)·내비게이션, 콘솔 컴포넌트)을 적시하라. "신뢰감을 주는 이유"를 한 줄로.

**E. 리프트 가능성·이식성 평가 (가장 중요 — 모든 후보에 강제)**
6. 각 후보마다: **① 라이선스**(MIT/Apache/BSD = 안전 차용 / AGPL·GPL = 경연 제출 제품 오염 위험 — 명확히 경고) ② **우리 바닐라-JS 스택과의 거리**(React/TS면 런타임 통째 리프트 불가) ③ **며칠 내 리프트 가능한 것**(디자인 토큰·CSS·레이아웃·IA·알고리즘·데이터 스키마) vs **레퍼런스 전용**(프레임워크 결합 런타임) ④ **현실적 이식 노력 S/M/L** ⑤ **핵심 리스크**.

**F. 덤**
7. 위 외에 **금융 특화 OSS 운영 콘솔**이나, "디자인시스템만 가져오고 운영 루프는 그 위에 재구현"한 성공/실패 사례·교훈이 있으면 추가하라.

### 출처 규칙 (반드시 준수)
- **공통 5가드**: ① 발견마다 출처 URL + 원문 짧은 인용 + 발행처·날짜 + 신뢰도(1차/2차/[미검증]). ② 핵심 주장 2출처 교차. ③ 링크 미접속 시 인용 금지. ④ 최신 우선(프레임워크는 빨리 바뀜 — 버전·날짜 명시). ⑤ 의심 시 보수적 [미검증].
- **라이선스는 사실**: 라이선스 주장은 레포의 **실제 LICENSE 파일/SPDX**에 근거하라. 확인 못 하면 [미검증]으로 표기 — 추정 금지. (경연 제출물의 라이선스 오염은 치명적.)
- **성숙도 사실**: 스타 수·최근 커밋·릴리스·프로덕션 사용 사례로 성숙도를 뒷받침(과장 마케팅 ≠ 성숙도).

### 출력 형식 (이 구조로 답하라)
1. **기능 후보 숏리스트 표** — | 프로젝트 | 한 줄 정체 | 운영계약 매핑(Case/Agent/Evidence/Approval/Audit 중 충족) | 라이선스 | 스택 | 성숙도 | 리프트 대상 | 이식난이도 S/M/L | 출처 |
2. **디자인 롤모델 표** — | 대상 | 정체 | 따라할 패턴(토큰/IA/컴포넌트) | 라이선스 | 바닐라-JS 이식성 | 신뢰감 이유 | 출처 |
3. **Top 3~5 추천** — 기능 1~2 + 디자인 1~2. 각: "왜 우리 롤모델인가 + **며칠 내 리프트 가능 범위** + 리스크".
4. **라이선스 경고 박스** — 차용 시 AGPL/GPL 오염 위험 있는 후보 명시 + 안전한(MIT/Apache/BSD) 대안.
5. **현실 점검(필수)** — 사용자 핵심 질문에 솔직히 답하라: **"며칠 안에 OSS급으로 작동" 중 무엇이 가능(리프트)하고 무엇이 불가(처음부터 재구현)인가.** 과약속 금지.
6. **근거표** — | 주장 | 출처 URL | 발행처·날짜·버전 | 신뢰도 | 원문 짧은 인용 | + **갭/[미검증]**.

### 딥리서치 깊이 지시
- 후보를 5개 이상 발굴하되, **각각을 이식성(E)까지** 끝까지 평가하라 — 평가 없는 나열은 가치 없음.
- 라이선스·스택·버전은 **레포 실물 근거**로. AGPL/GPL을 "그냥 오픈소스"로 뭉뜽그리지 마라.
- 우리 스택(바닐라 JS)을 잊지 마라 — "React 컴포넌트가 멋지다"는 우리에겐 *레퍼런스*일 뿐 *리프트*가 아니다. 둘을 분리해 표기.
- 제출 전 자체 점검: "이 후보의 무엇을, 어떤 라이선스로, 며칠 노력에, 우리 바닐라-JS/서버-API 스택에 실제로 옮길 수 있는가."

## Phase 2 — 로컬 티어다운 프롬프트 (소스 폴더 + 스크린샷 도착 후)

> 사용자가 OSS 폴더를 가져오고 캡처/이미지를 첨부하면, **builder(소스)·designer(디자인)** 에이전트에게 아래를 지시. Haiku는 기계적 인벤토리(파일트리·의존성·LICENSE 스캔·컴포넌트 목록)를 병렬로.

대상 레포: `<경로>` · 첨부 이미지: `<캡처들>` · 우리 앱: `02_제품/app/`(바닐라 JS) · 스타일: `styles.css`(8px 라운드, Pretendard).

1. **인벤토리** — 파일트리·LOC·의존성·**LICENSE/SPDX(차용 가능 여부 판정)**·스택. (Haiku)
2. **운영계약 매핑** — 레포의 어느 모듈이 우리 `Case/Agent/Evidence/Approval/Audit`를 구현하나. **file:line 인용.** (builder)
3. **리프트 가능성 매트릭스** — 모듈별 → `있는그대로 리프트 / 적응 후 리프트 / 레퍼런스 전용`, **이식노력**, **우리 쪽 타깃 파일**(어느 함수·CSS로 들어갈지). (builder)
4. **디자인 티어다운(스크린샷 기반)** — 컬러·간격·라운드·타이포 토큰 추출, 레이아웃 그리드, 컴포넌트 인벤토리, IA·내비 모델 → 우리 `styles.css` 토큰과 매핑하고 **충돌(8px 라운드·Pretendard·`verify_static.py` needle)** 표시. (designer)
5. **구체 리프트 플랜** — 리프트할 조각 **순서·일수 추정** 목록 + **"베끼지 말 것" 목록**(라이선스/스택 리스크). (builder+designer → orchestrator 통합)
6. **제약 리마인더** — `verify_static.py` 문자열 계약을 깨지 말 것 · 바닐라-JS 관용구 유지 · **PII 외부반출 경로 도입 금지** · 승인 게이트 우회 금지.

## Opus 점검 로그 (2패스)

### 패스 1 — 적대적 비판 → 보완
- **비판 ① "멋진 앱 구경"으로 흐를 위험.** 롤모델 리서치는 감탄만 하고 끝나기 쉬움 → 며칠 안 작동이라는 실무 목표와 단절.
  - **보완**: 모든 후보에 이식성(E) 평가 강제, 출력 5(현실 점검)에서 "리프트 가능 vs 재구현"을 명시 분리, 깊이지시에 "평가 없는 나열 무가치" 박음.
- **비판 ② 라이선스 오염 — 경연 제출물 치명타.** AGPL/GPL 코드를 무심코 차용하면 제출물 전체가 오염.
  - **보완**: 라이선스를 "사실(LICENSE 실물 근거)"로 규정, 미확인 시 [미검증], 별도 라이선스 경고 박스(출력 4), 안전 대안(MIT/Apache/BSD) 요구.
- **비판 ③ 스택 환상.** 폴리시드 OSS 대부분 React/TS인데 바닐라-JS에 "통째 리프트" 가능한 것처럼 착시.
  - **보완**: 스택 현실을 메타·역할문·깊이지시 3곳에 고정, "레퍼런스 vs 리프트" 분리 표기, 리프트 가능 대상을 디자인토큰·CSS·레이아웃·알고리즘·스키마로 한정.

### 패스 2 — 재비판 → 보완
- **재비판 ① 디자인 축이 paperclip과 중복되거나 충돌.** 이미 paperclip을 1차 본보기로 분석함.
  - **보완**: 경계 메모에 "B1 디자인 = paperclip 보강(추가 본보기·디자인시스템)"으로 역할 분리. Phase 2에서 우리 `styles.css` 토큰·`verify_static.py` needle과의 충돌을 명시 점검.
- **재비판 ② 핵심 차별(승인 게이트·PII 비반출)이 본보기 베끼다 훼손될 위험.** 외부 OSS엔 PII 외부전송·승인 우회 구조가 흔함.
  - **보완**: 하드룰을 메타·제약 리마인더에 박고, 그를 위반하는 구조는 본보기에서 제외하도록 명시.
- **재비판 ③ Phase 2가 막연하면 "폴더 받아도 뭐 할지 모름".** 사용자가 실제 폴더·이미지를 가져온다고 했으니 수신 즉시 실행 가능해야.
  - **보완**: Phase 2를 6스텝 체크리스트(인벤토리→매핑→리프트매트릭스→디자인티어다운→리프트플랜→제약)로 구체화, 각 스텝 담당 에이전트·산출 경로·file:line 인용 의무 명시.
