# D20 — AI 무기고: 특화 ML/DL 모델 · 시스템 정합성/이식성 · 도구 생태계(OSS·MCP·CLI)

## 메타 (프롬프트 밖 — 생성·운영용)
- **도메인**: LLM 에이전트를 **넘어선** 금융 AX의 "무기·방패". (무기) 금융 특화 ML/DL 모델, (방패) 레거시·코어뱅킹과의 **정합성·이식성**, (연장·탄약) OSS 스킬·MCP·플러그인·CLI **활용** + **JB 전용 사내 CLI/에이전트 하네스 구축**.
- **티어**: ★★★
- **심사 항목**: 3(AI Agent 설계·구현 — 에이전트가 휘두르는 비-LLM 역량과 통합) · 5(혁신·확장·운영리스크 — 레거시 이식·도구 생태계·자체 하네스의 확장성)
- **실행 배치**: 배치 3 (종합·차별 — D6·D9·D10 인용)
- **경계 메모**:
  - **D20 ↔ D9**: *무엇으로 돌리나(인프라·LLMOps·기술스택)* = D9 / *에이전트가 무엇을 무기로 쓰나(특화모델·도구·자체 CLI 구축 전략)* = D20.
  - **D20 ↔ D6**: 코어뱅킹·연동표준의 *목록·규격* = D6 / 그 위에서 AI를 *어떻게 정합·이식하는 패턴* = D20.
  - **D20 ↔ D10**: 보안·망분리·인젝션 방어 *기술* = D10 / 도구·OSS·MCP 결합의 *공급망·라이선스 리스크는 D20에서 짚되 보안 기술 구현은 D10 참조*.
  - **D20 ↔ D11**: 메모리·케이스 표현의 *학술 구조* = D11 / D20은 모델·도구·통합.

- **제품 연결**: JB LocalGuard OS의 에이전트(판단→행동초안→검증)는 LLM만이 아니라 **특화 ML/DL(신용·이상거래·시계열 부도예측 등)을 skill로 호출**하고, **기존 OSS·MCP·CLI를 합법·이식 가능하게 차용**하며, 필요 시 **JB 전용 사내 CLI/에이전트 하네스**를 구축한다. 핵심 메시지: *"하네스가 잘 되어 있으면 모델은 그 안에서 잘 작동한다."* (모델 성능은 하네스·도구·정합성이 좌우) — 운영계약 Case→AgentRun→Agent→Skill→Evidence→Approval→Audit의 **Skill 층을 특화모델·OSS·자체 CLI로 채우는 근거**.

## 딥리서치 프롬프트 (이 아래 통째로 붙여넣기)

당신은 **금융권 AI 시스템 아키텍트이자 도구·통합 전문 리서처**다. 목표는 "LLM 에이전트 **하나만**으로 금융 AX를 완성할 수 없다 — 에이전트는 **특화 ML/DL 모델(무기)**, **레거시와의 정합성·이식성(방패)**, **OSS·MCP·플러그인·CLI 같은 도구 생태계(연장)**, 그리고 **기업 전용 사내 CLI/에이전트 하네스**로 무장해야 비로소 현업에서 작동한다"는 명제를, **실제 금융권 사례·동료평가/기술 1차 자료**로 입증·구조화하는 것이다. 결과는 거버넌드 AI 에이전트 콘솔이 "LLM 래퍼"가 아니라 "특화모델·도구·자체 하네스를 오케스트레이션하는 운영체제"임을 정당화하는 데 쓰인다.

### 알아내야 할 것 (핵심 질문)

**A. 금융 특화 ML/DL 모델 (에이전트의 무기)**
1. 한국·글로벌 은행/금융이 실제로 운영하는 **금융 특화 ML/DL** 과제별 기법·대표 모델·성능: 신용평가·여신심사 스코어링, **이상거래탐지(FDS)·이상징후**, 부도/연체·조기경보 예측, 시계열 예측, 문서 정보추출(IE)·OCR, 추천. 각각 어떤 모델 계열(GBM/XGBoost, 딥러닝, GNN, 시계열 트랜스포머 등)이 표준이고 LLM 대비 왜 특화모델이 우월한가.
2. **LLM 에이전트가 특화 ML/DL 모델을 "도구(tool/skill)"로 호출·오케스트레이션**하는 설계 패턴: 에이전트가 언제 LLM 추론을 멈추고 특화모델 점수를 부르나, 결과를 어떻게 근거로 합성하나(hybrid neuro-symbolic / tool-calling). 대표 아키텍처·연구.
3. **한국어/한국 금융 도메인 특화 모델·데이터셋** 현황(금융 특화 LLM·임베딩·KO 금융 NLP 포함, 단 *모델 주권·로컬 호스팅 경제성은 D22 담당* — 여기선 "어떤 특화모델이 존재하나"까지).

**B. 시스템 정합성·이식성 (에이전트의 방패)**
4. **코어뱅킹·계정계/정보계·레거시(메인프레임·COBOL·배치)** 환경에 AI를 끼워 넣는 **통합 패턴**: API 게이트웨이, ESB/MSA, 어댑터·래퍼, 실시간 vs 배치, 데이터 파이프라인/CDC, 무중단·점진(strangler-fig) 도입. 금융권에서 검증된 패턴과 실패 사례.
5. **이식성·정합성을 높이는 설계 원칙**: 표준 오픈API·금융 공동 표준·이벤트 기반 아키텍처, 결합도 낮추기, 한 번 만든 에이전트/스킬을 다른 계열사·시스템으로 옮기는 portability(멀티-테넌시·구성 분리). "기존 시스템과 촥 맞아떨어지게" 만드는 구체 기법.

**C. 도구 생태계 — OSS·MCP·플러그인·CLI 활용 (연장·탄약)**
6. **에이전트가 외부 도구를 안전하게 결합**하는 아키텍처: 함수호출/tool-use, **MCP(Model Context Protocol)** 등 에이전트-도구 연결 프로토콜의 현황·성숙도, 플러그인·CLI 래핑. 대표 OSS 에이전트/오케스트레이션 프레임워크·스킬 생태계.
7. **OSS·도구 차용의 공급망·라이선스·보안 리스크**와 통제: 라이선스 적합성(상용·재배포), 의존성·공급망 공격면, 샌드박싱·권한 최소화. (보안 *기술 구현*의 깊은 부분은 D10 참조 — 여기선 도구 차용 관점의 리스크·통제까지.)

**D. 자체(사내 전용) CLI·에이전트 하네스 구축**
8. **기업이 자사 전용 내부 CLI/에이전트 하네스를 구축**한 사례·설계 원리: 왜 범용 도구 위에 사내 전용 CLI/하네스를 또 만드나(표준화·가드레일·반복작업 자동화·온보딩), 무엇을 표준화하나, 그리고 **"좋은 하네스가 동일 모델의 실효 성능을 얼마나 끌어올리나"**(스캐폴딩·컨텍스트·도구 접근이 모델 산출 품질에 미치는 효과)에 대한 근거.
9. 금융처럼 **규제·감사가 엄격한 환경의 전용 하네스 요건**: 모든 도구 호출·산출의 로깅·감사·승인 게이트를 하네스 레벨에서 강제하는 설계.

**E. (범용 덤 유도)** 위 외에 "에이전트의 무기·방패"로 볼 비-LLM 구성요소 — 규칙엔진·제약최적화·RPA·검색/랭킹·그래프 등 — 와 LLM의 **하이브리드** 패턴 중, 금융 AX에서 차별을 만드는 것이 있으면 추가 카테고리화하라.

### 패스 1 — 적대적 비판 → 보완
- **비판 ① 기존 D9·D10·D6·D11과 광범위 중복.** "통합·도구·모델"은 네 도메인과 겹쳐 산출이 흐려진다.

- **비판 ② 성능 수치 환각 자석.** "특화모델이 N배 정확" "MCP가 표준" 류를 벤더 블로그가 부풀린다.
  - **보완**: 도메인 가드에 "성능=벤더=2차/미검증, 1차=논문·공식 벤치마크", "MCP·신생 프로토콜은 공식 스펙으로만 정의"를 박고 출력 6에 한계 표 의무화.
