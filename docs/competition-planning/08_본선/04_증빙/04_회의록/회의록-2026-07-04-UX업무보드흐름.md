# 회의록-2026-07-04-UX업무보드흐름 (전략회의 2)

> ⚠️ 이 회의는 스코프 결정([[회의록-2026-07-04-스코프확정]])이나 시연 범위([[회의록-2026-07-04-시연기술범위]]) 회의와 별개 — **RM 업무보드(Kanban)의 화면·인터랙션 흐름을 손그림+구두로 즉흥 시연·논의**한 세션이다.

## 참석 / 맥락
- 팀 **망상궤도**: 김주용 · 김민주(디자인·UX) · 이승보(개발).
- 김민주가 손그림/구두로 업무보드 UX 흐름을 설명하고, 김주용·이승보가 질문·보완하며 상세화. 회의 종료 직후 김민주가 **8분 만에 Figma로 시각화**해 팀에 공유(아래 "연결" 참고) — 이 회의록의 시각 자료 원본.

## 핵심 요약 (3줄)
1. **업무보드 인터랙션 확정**: 케이스 리스트(우선순위 정렬) → 클릭 → 배정 에이전트 나열 → 스킬/MCP/내부규정으로 처리 → 개별 MD → 취합 에이전트가 최종 MD 하나로 통합 → **키보드(숫자·방향키·엔터·스페이스)만으로 승인** → 실제 업무 수행 → 케이스 종료.
2. **상태 표시 원칙**: 케이스는 **전/중/후** 카운트로 표시(총 건수 아님), 로딩은 %없이 스피너만, 완료 항목은 딤드 처리 + 알림(뱃지)으로 복귀 유도.
3. **결과물 뷰어 방향**: 에이전트별 MD를 탭 vs 플로팅 오버레이 중 논의 → **노션/옵시디언 스타일 MD 뷰어 + 파일간 위키링크 내비게이션**으로 수렴, Figma 와이어프레임 3슬라이드로 즉시 실증.

## 발췌 — 주제별 서술 (프롬프트 인풋용 줄글)
> 핑퐁(턴바이턴 대화체)이 아니라, 흐름을 아는 사람이 읽듯 주제별로 서술한 것 — 원문(FULL.txt) 내용을 축약 없이 담았다.

**메인 화면 골격 — 케이스 리스트 → 클릭 → 에이전트 배치 (00:05~01:51)**
김주용이 17:27에 "대시보드 UX/UI 논의"를 시작한다고 알리며 즉흥 설명 방식("제가 알려드릴 수가 없을 거고요")으로 진행을 예고했다. 김민주는 메인·서브 화면 구조를 설명했다 — 메인은 한눈에 보이고 서브는 스크롤해야 내려가는 구조이며, RM이 로그인하면 "오늘 해야 할 일"(오늘 처리할 케이스 개수)을 먼저 보고, 케이스가 여럿이면 **우선순위**로 정렬된다. 첫 케이스를 클릭하면 하단에 배정된 에이전트들이 "쫘르륵" 나열되고, 담당자가 서브 케이스에서 엔터를 치면(승인) 해당 케이스가 로딩 상태로 전환되며 담당자는 곧바로 다음 케이스로 넘어간다. 로딩이 끝나면 에이전트들이 결과를 MD 파일로 "아주 깔끔하게" 정리해 놓고, 담당자가 다시 엔터를 여러 번 눌러("너 통과, 너 통과, 통과, 통과") 각 에이전트 결과를 승인하면 최종 파일 하나가 만들어진다.

**에이전트 구조 — 역할 중심, 취합 에이전트 (02:04~04:11)**
김주용이 케이스 옆의 에이전트 구조도를 설명했다. "여기 에이전트가 하나 있을 거고 — 에이전트 A. 얘가 이 케이스를 담당하겠죠. 저희는 역할 중심이니까"라며, 각 케이스 = 각 담당 에이전트라는 구조를 확인했다. 여러 에이전트가 나오면 각각 스킬·MCP·플러그인·내부 규정을 가지고 케이스에 접근하며, 모아진 결과물은 별도의 **취합 에이전트**가 "최종 MD"로 통합한다(결과물은 하나일 수도 두 개일 수도 있음). 담당자는 그 결과물을 근거로 실제 업무(전화, 수기 보고서, 각종 신청 등)를 수행하고, "이 근거들을 바탕으로 어프로브, 오케이 너 잘했어" 하면 케이스가 닫히고, 못 했으면 다시 돈다.

**승인 인터랙션 — 엔터의 책임 분산, 환각 방지 (04:26~06:19)**
김민주는 각 에이전트 항목에 "이걸 분석하려고 하니까 승인해 주세요" 같은 한 줄 설명과 승인 버튼이 있고, "이건 좀 아닌 것 같은데" 싶으면 웬만하면 승인하지 않는 흐름을 설명했다. 김주용은 인터랙션이 클릭이 아니라 "엔터 혹은 방향키·스페이스바"라고 못박았고, 김민주는 "엔터 하나에 너무 큰 책임을 주면 안 된다"고 지적했다(→ 이후 [[회의록-2026-07-04-스코프확정]]에서 오승인 방지용 "스페이스 2회" 컨벤션으로 정착). 이승보는 승인 시 화면이 전환되는 트랜지션과 함께 에이전트가 배정되고, 그 에이전트들이 케이스를 분석해 하위 에이전트들이 만들어 둔 스킬을 불러와 처리한 뒤 MD 파일을 생성하고, 최종적으로 하나의 MD 파일로 취합해 RM이 다음 업무를 진행하게 한다고 설명을 보탰다. 김주용은 승인 여부에 따라 "마음에 들면 오케이 잘했어, 마음에 안 들면 채찍 들고 다시 해"라는 비유를 들었고, 김민주는 결과에 수정이 필요할 수도 있다고 덧붙였다. 이어 김주용은 잘못된 수치·출처(환각) 문제를 줄이는 방법으로 **RAG 같은 내부 지식 데이터**나 **API로 가져온 실제 데이터**를 근거로 삼고 그 위에 "하네스"(승인 게이트)를 씌우는 것이 가장 효과적이라는 데 공감대를 이뤘다.

**키보드 전용 네비게이션 상세 (06:36~07:52)**
김주용은 맥북 유저 기준으로 키보드 방향키(상하좌우)로 이동하고 스페이스로 세부 정보를 토글하는 방식을 제안하면서도, 정보가 너무 많아질 수 있다는 우려를 함께 언급했다. 엔터를 연속으로 눌러 다음 항목으로 넘어가되 잘 모르는 항목은 화살표로 건너뛰고, 처리 중인 항목은 불투명/희미하게 표시하며 로딩 애니메이션을 준다는 아이디어도 나왔다. 김민주는 엔터를 누르면 곧바로 로딩이 나와야 하고 **퍼센트 진행률 표시는 불필요**하며 그냥 로딩 인디케이터만 있으면 충분하다고 정리했다. 김주용은 서브 화면 = 케이스 서브 에이전트 + 다양한 뷰 페이지들이라는 계층 구조를 재확인했다.

**전/중/후 카운트 표시 (07:53~09:33)**
김민주는 케이스 상태 표시를 총 건수가 아니라 **전/중/후** 3분류 카운트로 하자고 제안했다(예: 총 7건 중 진행전 5·진행중 1·진행후 0). 김주용과 이승보는 각 상태를 클릭하면 해당 상태의 케이스 목록으로 드릴다운할 수 있어야 한다고 확인했다. 김민주는 처리 완료된(자동화) 항목은 그냥 결과만 보여주면 충분하고, 항목을 누르면 에이전트별 전/중/후 상태가 한 단계 더 나온다고 설명했다. 김주용은 에이전트/API 레이어와 전반적 상태 레이어를 계층으로 나눠 헷갈리지 않게 하고, 다양한 뷰(전세 관련, 운영 스킬, MCP 사용 이력 등) 중에서도 핵심적으로 작동하는 것과 차별성 있는 것만 선택해서 보여주자고 정리했다.

**노트북 화면 잘림 + 업무보드 버튼 + 토큰 비용 우려 (09:54~11:05)**
김민주는 화면 하단을 딤드(블러) 처리하고 화살표로 "여기가 노트북 잘리는 부분"임을 표시하자고 했다. 담당자(예: RM 1명)가 로그인하면 "업무 보드" 버튼이 크게 있고 뷰 버튼은 2개 정도면 충분하다고 정리했고, "왜 이렇게 15개나 있네" 하며 케이스를 훑어보다 마우스 또는 키보드로 시작하는 흐름을 시연했다. 이 대목에서 김주용은 토큰을 많이 쓸 것 같다는 우려("토큰 진짜 많이 써요")를 농담 섞어 언급했고, 이승보는 "이거를 위해서 아직 코덱스, 플로우도 안 쓰고 있다"며 당시 미사용 상태임을 확인했다 — 개발 리소스·비용에 관한 실무 코멘트로, 제품 스펙 결정 사항은 아니다.

**확장성·납품 가능성 논의 (11:18~12:55)**
계열사 확장 관련해 김주용의 "기부 계열사지?" 등 짧고 문맥이 이어지지 않는 발화가 있었고, 김민주는 이 제품이 "금융에서도 안 써도 되고 다른 곳에서 그냥 저희 과제에서 쓰면 될 것 같다"며 비금융 확장 가능성을 언급했다. 다만 김주용은 행정처리 업무는 한글 문서·기존 전산시스템에 의존하기 때문에 확장이 제한적이라는 한계도 함께 짚었다. 김주용은 "이게 시스템 안으로 들어오면 사실상 납품할 수 있는 제품"이라는 자체 평가를 내렸고, 김민주는 "이것만 되면 무조건 3점"이라고 응답했다(심사 배점에 대한 근거 없는 구두 자평). 이어 김주용은 LLM 호출 시 딜레이가 있어 바로 처리되지 않고 "기다려" 상태가 되며, 완료되면 알림(체크·벨)이 뜨고 사용자가 그걸 보고 돌아와 확인·승인한 뒤 다음 순번으로 넘어가는 순차 처리 흐름을 설명했다.

**우선순위 기준(미정) + 손그림 활용 자평 (13:04~13:46)**
이승보가 케이스 우선순위 산정 기준을 먼저 잡아야 한다고 문제를 제기했고, 김주용은 "기준은 근거"라고만 답해 구체적인 산정 로직은 이 회의에서 확정되지 않았다. 이후 손그림 스케치로 설명한 방식에 대해 김주용이 "역시 손그림이 짱이에요", 김민주가 "이게 역시 개발자랑 다 같이 읽어야 돼요"라고 긍정적으로 평가했다.

**케이스 카드 레이아웃(가로형) + 서브페이지 전환 (13:46~15:15)**
김민주는 케이스를 세로로 쌓기보다 **가로형 카드**로 여러 개(케이스 1·2·3·4…) 나열하는 레이아웃을 선호한다고 밝혔다. 서브 페이지에 진입하면 해당 카드가 스트로크(테두리) 처리되며 두꺼워지는 트랜지션을 준다고 했고, 은행/도메인 표시(전세 등 예시로 언급), 소요 시간 표시는 "있으면 좋지만 핵심은 아니"라고 우선순위를 정리했다.

**에이전트 카드 상세 — 이유·기대값·데이터출처, 승인 애니메이션 (15:21~16:43)**
김민주는 에이전트 카드의 구성 요소를 정리했다 — 이 에이전트가 붙는 **이유**, **기대값**(무엇을 예측/분석할지), 그리고 "엔터를 눌러 실행하세요"라는 승인 CTA. 승인(엔터)하면 카드가 스트로크 처리 후 딤드되며 아래로 내려가고 다음 카드가 위로 올라오는 트랜지션이 이어지며, 각 카드는 어떤 데이터를 근거로 분석할지와 기대값 예측을 함께 보여준다. 이승보는 처리 완료 느낌을 명확히 표현해야 한다고 덧붙였다.

**키보드 전용 재확인 + 완료 알림 플로우 (16:48~19:11)**
김민주는 사용자가 승인하고 싶지 않은 항목은 그냥 두면(보류) 된다고 정리하고, "절대 마우스 사용 금지"라는 원칙을 재차 강조했다 — "한 손에는 노트를 해야 되고 한 손에는 이건 움직여야 돼"라는 이유였다. 숫자키(예: 2)만 눌러도 케이스를 열람할 수 있어야 한다고 했다. 처리가 완료되면 상단에 "승인하신 것은 다 완료됐다"는 알림이 뜨고, 클릭하면 업무보드로 돌아가 케이스별 전/중/후 현황을 다시 확인한다(예: 케이스1 완료·어두운 색 표시, 케이스2 진행중, 나머지 4개 대기).

**결과물 표시 방식 — 통합 vs 개별, 탭 vs 플로팅 (19:21~20:59)**
완료 후에는 "결과물"과 "사용한 스킬"이 표시되며, 스킬·출처 아이콘(봉투 모양)이 하단에 나열된다. 이 결과물을 통합해서 보여줄지 개별로 보여줄지에 대한 논의가 이어졌다 — 에이전트별 MD가 있고 이를 누르면 통합본에서 보거나 개별 파일로 들어가 볼 수 있다는 아이디어였다. 표시 방식 후보로 **탭**(김주용 제안, 김민주 동의)과 **플로팅 오버레이**(이승보 제안 — "화면만 위에 이렇게 띄우게", 배경 딤드 + 중앙 모달)가 논의됐다.

**최종 결론 — 노션/옵시디언 스타일 MD 뷰어, Figma로 즉시 시각화 (21:03~22:37)**
논의는 "PDF 같은 거 볼 때처럼" 노션 MD 파일처럼 예쁘게 정리하고 아래 선을 눌러 링크를 타고 이동하는 방식으로 수렴했다. 김주용은 통합본 MD 안에 개별 MD 파일들이 **링크**로 연결돼 있어야 하며, "옵시디언처럼, 노션처럼" 눌러서 타고 들어가면 해당 MD를 볼 수 있어야 한다고 정리했고, "@에이전트" 멘션으로 그 에이전트 화면으로 이동하는 아이디어도 논의됐다(기술 구현 가능성은 미확정). 회의 말미에 김민주가 "이거 두 개만 이렇게 만듭니다... 그리고 이제 이거를 빨리 8분 동안 피그마로 만들어서 제가 보내드릴게요"라고 마무리했고, 실제로 회의 직후 8분 만에 Figma 와이어프레임 3슬라이드로 시각화해 팀에 공유했다(node 4924:2701).

## 공유·결정 (구조화 요약)

1. **메인 화면 골격**: RM이 로그인하면 "오늘 해야 할 일"(케이스 리스트, 우선순위 정렬)을 먼저 본다. 첫 케이스를 클릭하면 하단에 **배정된 에이전트들이 나열**된다. 메인/서브 화면을 나누고 서브는 스크롤.
2. **에이전트 구조 (역할 중심)**: 케이스마다 담당 에이전트가 있고("에이전트 A"), 저희 제품은 **역할 중심** 설계다. 각 에이전트는 보유한 **스킬·MCP·플러그인·내부 규정**을 활용해 케이스를 처리하고 개별 MD 결과물을 만든다. 여러 에이전트의 결과물을 **별도의 취합 에이전트**가 최종 MD 하나(또는 소수)로 통합한다. 담당자는 그 결과물을 근거로 실제 업무(전화·수기보고서·신청 등)를 수행하고, 승인하면 케이스가 닫히고 미승인이면 재작업으로 돈다.
3. **승인 인터랙션 = 키보드 전용**: 마우스 클릭이 아니라 **엔터 / 방향키 / 스페이스바**. "엔터 하나에 너무 큰 책임을 주면 안 된다"는 문제의식이 나왔고(→ 이후 [[회의록-2026-07-04-스코프확정]]에서 **오승인 방지용 "스페이스 2회" 컨벤션**으로 확정됨), 방향키+스페이스로 세부정보를 토글하는 방식도 논의됐다.
4. **환각 방지 원칙**: 스스로 판단하지 못해도 **RAG형 내부 지식 데이터**나 **실제 API로 가져온 데이터**를 근거로 삼고, 그 위에 승인 게이트("하네스")를 씌우는 것이 환각을 가장 줄이는 방법이라는 데 공감대.
5. **케이스 상태 표시**: 총 건수가 아니라 **전/중/후** 3분류 카운트로 표시(예: 총 7건 중 진행전 5·진행중 1·진행후 0/1). 각 상태를 클릭하면 해당 상태의 케이스 목록으로 드릴다운. 처리 완료(자동화) 항목은 그냥 결과만 보여주면 충분하다는 의견. 로딩 중인 항목은 불투명/희미 처리 + 로딩 애니메이션, **퍼센트 진행률 표시는 불필요**(스피너로 충분).
6. **케이스 카드 레이아웃**: 세로 나열보다 **가로형 카드**(Case 1·2·3·4…)를 선호. 서브페이지 진입 시 해당 카드가 스트로크(테두리) 처리되며 두꺼워지는 트랜지션. 손그림 스케치로 설명한 것에 대해 "역시 손그림이 짱"이라는 팀 반응(개발자와 함께 보기 좋은 매체로 재확인).
7. **에이전트 카드 상세 구성**: 각 에이전트 카드는 (a) 이 에이전트를 쓰는 **이유**, (b) **기대값**(무엇을 예측/분석할지), (c) **소요 시간**(N분), (d) **데이터 출처**, (e) **예상 산출물(.md)** 을 표시하고 "Enter를 눌러 승인해주세요" CTA로 마무리한다. 승인 시 카드가 스트로크 처리 후 딤드되며 아래로 내려가고, 다음 카드가 위로 올라오는 트랜지션(순차 처리 시각화).
8. **완료 알림 플로우**: 처리 완료 시 상단에 알림(예: "승인하신 것은 다 완료됐다")이 뜨고, 클릭하면 업무보드로 복귀해 케이스별 전/중/후 현황을 재확인한다. 완료 케이스는 어두운 색으로 표시.
9. **결과물 뷰어 — 탭 vs 플로팅, 최종 방향**: 에이전트별 MD를 통합해서 보여줄지 개별로 보여줄지 논의 끝에, **탭**(김주용 제안)과 **플로팅 오버레이**(이승보 제안 — 배경 딤드 + 화면 중앙에 MD만 띄우는 모달형) 두 안이 나왔고, 최종적으로는 **노션/옵시디언처럼 예쁘게 정리된 MD + 파일 간 위키링크 내비게이션**(통합본 MD 안에서 개별 에이전트 MD로 링크 타고 들어가는 구조, "@에이전트" 멘션으로 해당 에이전트 화면 이동 아이디어도 논의)으로 수렴 — Figma 시안(슬라이드 2)에서 플로팅 다크 오버레이 + MD 카드로 실제 구현됨.
10. **제품 완결성·확장성 자평**: "이게 시스템 안으로 들어오면 사실상 납품할 수 있는 제품"이라는 자체 평가. 비금융 도메인으로도 확장 가능하다는 의견이 나왔으나, 한글 문서·기존 전산시스템에 의존하는 행정 업무는 확장이 제한적이라는 한계도 함께 지적됨(참고용 논의, 확정 아님).
11. **케이스 우선순위 기준은 미정**: 이승보가 "우선순위 기준을 먼저 잡아야 되겠다"고 문제 제기했고, 김주용은 "기준은 근거"라고만 답함 — 구체적인 산정 로직(마감일·금액·리스크등급 등)은 이 회의에서 확정되지 않음. 후속 설계 시 별도 결정 필요.
12. **손그림 스케치 방식에 대한 팀 반응**: 김민주가 손그림/구두로 설명한 방식을 김주용이 "역시 손그림이 짱이에요", 김민주가 "이게 역시 개발자랑 다 같이 읽어야 돼요"라고 긍정 평가 — 이후에도 UX 논의는 손그림 스케치 우선으로 진행하는 것이 팀 관행으로 재확인됨.
13. **토큰/비용 우려(구두, 미정)**: 김주용이 "토큰 진짜 많이 써요"라고 우려를 표하자 이승보가 "이거를 위해서 아직 코덱스, 플로우도 안 쓰고 있다"(당시 미사용)고 답함 — 개발 리소스/비용 관련 실무 코멘트로, 제품 스펙 결정은 아니다.
14. **심사 배점 "3점" 자평(근거 없음)**: 김민주가 "이것만 되면 무조건 3점"이라고 언급 — 실제 심사 기준에 근거한 수치가 아닌 구두 자평이므로 제출 문서에 인용 금지.
15. **계열사 확장 관련 스치는 언급**: 김주용의 "기부 계열사지?" 등은 문맥이 이어지지 않는 짧은 발화로, 이 회의에서 계열사 확장 방향이 별도로 확정되지는 않음(관련 결정은 [[회의록-2026-07-04-스코프확정]] 등 다른 회의 참고).
