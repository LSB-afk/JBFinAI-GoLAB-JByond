# 09 — 119 에이전트 확장 설계도: LLM 라우팅 관측·복구 (Q14 담당)

> **상태**: [[03-119-사고대응-에이전트]]의 관할 확장안. 폴백 사다리·구조화 오류 로그·사람 큐 격상 응답은 **[E4 구현됨]**(`api-proxy.mjs`), 119 엔티티·사고 티켓 UI는 [분기/미확정].
> **담당 문제**: [[Q14-오류로깅-폴백사다리]] — "라우팅 오류를 어떻게 알고, 어떻게 복구하나?"
> **위치**: 119는 새 에이전트가 아니다. 기존 119(Kill Switch·Rollback·Replay·Quarantine·Hotfix)의 **감지 입력에 LLM 라우팅 실패 스트림을 편입**하는 것이 이 문서의 전부다. 별도 에이전트를 늘리지 않는다.

## 역할 한 줄 (확장분)

게이트웨이의 오류 스트림(`errorClass`)을 상시 구독해 **1차 복구는 폴백 사다리에 맡기고(자동·이미 구현), 사다리로 못 잡는 패턴 — 반복 실패·격상 급증·조용한 품질 저하 — 을 사고로 승격**시키는 119의 관측 전초.

## 감지 입력 확장 (기존 119 상태기계에 연결)

```
llm-runs.jsonl (E4) ──┐
  errorClass 스트림    ├─→ [Detect] ─ 기존 119 상태기계 그대로
guardrail 이벤트 ──────┘      (Hold / KillSwitch / Quarantine / Replay / Rollback / Hotfix)
```

| 신호 | 임계 (초안, [가정]) | 119 반응 |
|---|---|---|
| 단발 실패 | `retry<2` 폴백 성공 | 반응 없음 — 사다리가 처리 (E4) |
| `escalated: true` | 발생 즉시 | 사람 큐 티켓 생성 + 해당 Case 자동진행 Hold |
| 동일 `errorClass` 반복 | 10분 내 3회 | Hold + 해당 엔진 티어 일시 강등 제안 |
| `timeout` 급증 | 시간당 5회↑ | Quarantine 후보(엔진·스킬 격리) + Hotfix 제안 |
| 조용한 실패 (오류 0인데 품질 하락) | Verification 불일치율·사람 반려율 상승 [설계] | Replay로 재현 → 사고 티켓 |

## 에이전트 폴더 파일셋 (paperclip式)

```
agents/119/
├── agent.yaml      # 기존 119 설정에 라우팅 관측 wakeup 추가
├── SOUL.md         # 확장 지침 (아래 §를 기존 SOUL에 append)
├── tools.json      # ledger_tail·usage_report 구독 추가
└── memory.md       # Incident Memory 카드 (재발방지 기억)
```

### agent.yaml (확장 diff만)

```yaml
heartbeat:
  intervalSec: 300            # 5분 순찰 — 오류 스트림은 신선도가 생명
  wakeOnAutomation: true      # escalated:true 발생 시 즉시 wakeup [설계]
adapter: claude_local         # 사고 판단은 프런티어 티어 (오판 비용 > 토큰 비용)
timeoutSec: 180
```

### SOUL.md 확장 § (기존 119 SOUL에 이 절을 추가)

```markdown
## LLM 라우팅 관측 (확장 관할)

너의 감지 입력에 게이트웨이 원장이 추가됐다. 매 wakeup마다:
1. `ledger_tail`로 직전 순찰 이후 레코드를 읽는다.
2. errorClass별 집계 후 임계표와 대조한다. 임계 미만이면 아무것도 하지 않는다 —
   폴백 사다리가 이미 복구한 실패에 사고 티켓을 만드는 것은 소음이다.
3. 임계 초과 시에만 사고 티켓을 연다: {신호, 근거 runId 목록, 영향 Case 목록,
   권고 조치(Hold/강등/Quarantine), 사람이 결정할 것 한 줄}.
4. escalated:true 건은 무조건 사람 큐 티켓으로. 요약에 "AI 두 엔진 모두 실패,
   수동 처리 필요"와 원문 프롬프트의 caseId(내용 아님)만 적는다.

## 금지선 (확장분)
- 엔진·스킬을 직접 끄지 않는다. Kill Switch는 critical(PII 반출·대량 실행)에만,
  그 외는 전부 제안 → 사람 승인.
- 사고 티켓에 프롬프트 원문·고객 정보를 복사하지 않는다. runId 포인터만.

## 기억 규칙 (Hermes식 증류)
- 모든 오류를 기억하지 않는다. Incident Memory 카드 승격 기준:
  (a) 사람 승인까지 간 사고 (b) 동일 원인 2회 이상 재발 (c) Hotfix로 이어진 건.
- 카드 형식: 증상 → 원인 → 재현 절차(Replay 파라미터) → 조치 → 재발 여부.
  다음 사고 때 이 카드부터 검색한다 — 축적될수록 진단이 빨라지는 지점.
```

### tools.json (확장분)

| 도구 | 바인딩 | 상태 |
|---|---|---|
| `ledger_tail` | `llm-runs.jsonl` 증분 읽기 (errorClass 필터) | **E4** (파일 존재) |
| `usage_report` | `GET /llm/usage` — totals.errors·escalated 카운터 | **E4** |
| `incident_ticket` | 사고 티켓 생성 → 콘솔 승인 큐 노출 | [설계] |
| `case_hold` | 해당 Case 자동진행 보류 플래그 | [설계, moveCaseToColumn 연동] |
| `replay` | 기록된 runId 파라미터로 재실행 | [설계, /llm 재호출로 구현 용이] |

## 메모리 계약

| 계층 | 읽기/쓰기 | 내용 |
|---|---|---|
| Incident Memory | 쓴다 (주 저장소) | 사고 카드 — 증상·원인·재현·조치·재발 |
| Agent Memory | 쓴다 | 불안정 엔진·스킬 표시 ("codex exec, 긴 프롬프트에서 timeout 경향, n회") |
| Case Memory | 읽기만 | 영향 Case 식별 (caseId 코드만) |

## 승인 게이트

- **자율**: 관측·집계·티켓 발행·escalated 사람 큐 전달.
- **제안→승인**: 티어 강등, Quarantine, Hotfix.
- **사람만**: Kill Switch 해제, 사고 종결 선언.

## 구현 리얼리티

| 구간 | 내용 |
|---|---|
| 지금 (E4) | 폴백 사다리·시도별 JSONL·escalated 응답·오류 분류 6종·토큰 패널 오류/격상 카운터. 데모: `timeoutMs:1` 호출로 사다리 라이브 시연 |
| MVP 추가 | escalated 건을 승인 큐 항목으로 렌더(1 UI 연결 — Q14 "남은 것"과 동일) |
| 로드맵 | automation wakeup·Replay 도구·Verification 불일치율 드리프트 지표 |
