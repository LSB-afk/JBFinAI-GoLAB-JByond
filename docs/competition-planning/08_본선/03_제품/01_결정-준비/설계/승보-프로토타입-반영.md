## 3. 하네스 시스템 계약 (`harnessCore.js` / `harnessRegistry.js` / `harnessVerification.js`)

- **harnessCore.js** — business 로직 0줄. manifest 등록소(`registerHarness`/`getHarness`/`listHarnesses`), hook 실행기(`harnessRunHooks`, 위반 로그 최근 200건 보관), 공통 guardrail 검사 유틸 4종(`harnessGuardCheckPII`/`harnessGuardCheckAssertions`/`harnessGuardCheckScope`/`harnessGuardCheckAutoClose`/`harnessGuardCheckApprovalRequired`), 신규 역할 스캐폴드 기술자 `describeNewHarnessScaffold(roleKey)`.
- **harnessRegistry.js** — 두 하네스(jeonse-protection / jb-woori-capital)를 표준 manifest로 등록만 한다. manifest는 `id·kind·scopeKey·scopeValue·displayName·routeBase·sidebarConfig·countService·searchService·caseCreationFlow·agents·skills·commands·hooks·rules·guardrails·verification`(16키). **main/default/safety 이름의 alias 하네스는 존재하지 않음** — 만들면 verify 실패.
  - jeonse-protection: `enforceHooks: true`, requiredHooks 3종(beforeCaseCreate/beforeAgentRun/beforeCustomerMessage), requiredAgents 10, requiredCommands 6.
  - jb-woori-capital: `enforceHooks: false`(**TODO 명시** — "hook 연결은 다음 단계"), requiredAgents 13, requiredCommands 0. → JBWC는 전세보호보다 하네스 성숙도가 한 단계 낮다(hook 미배선).
- **harnessVerification.js** — ECC의 Smoke Test/AgentShield 관점을 6개 검증기로 구현: `verifyHarnessIntegrity`(manifest 필수키+count/search 함수+전용wizard+skills≥1) · `verifyRoleHarnessScope`(scopeProbe: 미지정 조회 시 예외 발생 확인) · `verifyNoForbiddenRoleResurface`(삭제 역할/광주은행 레일/label-only 심볼 재유입 검사) · `verifyNoPIILeakage`(저장소 직렬화+DOM innerText 스캔) · `verifyAgentRegistryCompleteness`(agent 수 + 11개 필드 + 공통 blockedActions 전원 포함) · `verifyHookCoverage`(enforceHooks 시 hook/command 연결 확인). `runHarnessSelfTest(harnessId)`가 6개를 집계해 `window.__lastHarnessSelfTest`에 저장 — 브라우저 콘솔/명령(`/jeonse-run-smoke-test`)에서 실행 가능한 **런타임 자체 검증 루프**.

## 4. ROLE_HARNESS_CONTRACT / SECURITY_GUARDRAILS 핵심 규칙

### 4.1 ROLE_HARNESS_CONTRACT.md
- 기준 구현은 `jeonse-protection`. **템플릿 복사-붙여넣기 alias 금지** — 신규 역할은 `describeNewHarnessScaffold`가 기술한 9종 파일을 각자 새로 작성.
- 신규 역할 추가 8단계: scaffold 체크 → 전용 파일세트(config→agents+skills→rules+hooks→scope강제db→services→helpers/views→commands→sidebar→router) → index.html 로드순서(core→config→registry→rules→db→services→views→commands→sidebar→router) → harnessRegistry 등록(enforceHooks:true, requiredHooks≥3) → app.js 배선 → verify_static.py 갱신 → e2e+smoke 추가 → 문서 갱신.
- 금지 alias 패턴: 타 하네스 business 함수/상수 참조(`jbwcTable(`/`JBWC_*`/`jpoTable(`/`JPO_*`(자기 외)/메인 `agents` 배열/`roleDashboardPage(`) · `mainHarness`/`defaultHarness`/`safetyHarness` alias · scope 없는 전체조회 · label-only 대시보드 회귀 · 삭제 축 재유입(`consumer-protection-dashboard`/`compliance-dashboard`/광주은행).
- **다음 역할 확장 후보로 문서가 이미 제안**: RM(scope `rm-officer`, 히어로 전주카페 seed 승격), 기업여신 담당자(전결 L1~L4), 보이스피싱/FDS 담당자(기존 label형 fds-dashboard를 이 계약으로 승격), JB우리캐피탈(hook 배선 남음).

### 7.6 ⚠️ 히어로 케이스 불일치 [미결/7-4]

- **canon/예선 앱**: 히어로 `JBG-104`(전주 중앙로 카페, `?demo=sme`, 메인 `app.js` 14-에이전트 레이어, owner="Cashflow Triage Agent").
- **승보 신규 4콘솔**: 별도 히어로 격 케이스 `CCL-0001`("전주 카페 운전자금 검토", corporate-credit 콘솔, 8-에이전트 별도 레이어) — 지명(전주 카페)만 겹치고 코드·데이터·에이전트 세트는 완전히 다른 케이스.
- 두 "전주 카페" 케이스가 같은 발표에 동시에 등장하면 심사자에게 "같은 케이스를 두 번 다르게 보여준다"는 혼선을 줄 수 있음. **데모 정본을 하나로 택할지(JBG-104 유지 + CCL-0001은 확장성 증거로만 언급 / 또는 역할축 4콘솔을 메인 데모로 승격하고 JBG-104는 레거시로 격하), 아니면 의도적으로 "메인 콘솔"과 "역할별 신규 콘솔"을 별개 레이어로 병행 시연할지는 미결정** — [미결/7-4], 팀 결정 필요.
