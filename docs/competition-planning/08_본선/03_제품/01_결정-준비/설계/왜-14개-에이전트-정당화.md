# 왜 정확히 14개 에이전트인가 — 바텀업 정당화

> **목적**: "탑다운으로 그럴듯하게 숫자를 고른 것 아니냐"는 깐깐한 심사위원 질문에 **상향식 기능분해**로 답한다. 하향식 정당화(먼저 14를 정해놓고 이유를 채우는 것) 금지 — 아래 §2는 스코프·규제·데이터만 놓고 독립적으로 다시 센 결과다. 결론이 14가 아니면 정직하게 그렇게 쓴다.
>
> **근거 범위(창작 금지, 이 파일들만 인용)**: `_canon.md` §2(14 로스터)·§4(규제 인용)·§8(운영계약), [[키스톤-확정]], [[키스톤-역할축-검증]], `설계/agents-에이전트 아키텍처 재설계(뽀족한 전문가 로스터)`, `설계/agents-v2-paperclip기반-재설계`, `04_증빙/02_분석자료/리서치/02_결과-원문/`의 D8·D9·D14·D15-결과.md, D16(키스톤 문서를 통해 재인용).
>
> **선행 문서와의 관계**: `agents-에이전트 아키텍처 재설계`와 `agents-v2-paperclip기반-재설계` 두 문서 모두 "14 유지"로 이미 결론 내렸다. 하지만 두 문서의 논증은 "5개 직군은 너무 굵고 20개는 과하다, 그 사이 어딘가"라는 **구간 논증**이지, 왜 정확히 14인지를 단위 하나하나 세어 보인 **개수 논증**은 아니다. 이 문서는 그 공백을 메운다. 결과적으로 같은 14에 도달하지만, 도달 경로가 다르다 — 그리고 §4에서 기존 두 문서가 언급하지 않은 조건부 불일치(사후관리 편입 시 15)를 새로 드러낸다.

## 0. 결론 먼저

- **확정 3도메인(여신·전세·피싱) + 오케스트레이션 + 통제기능 스코프에서, 독립적으로 다시 센 값도 14다.** §2의 분해가 canon §2의 14종과 1:1로 재현된다 — 우연이 아니라, 동일한 운영계약·동일한 규제환경·동일한 실제 앱(app.js) 구현을 다른 방향(기능→개수)에서 세었기 때문이다.
- **단, 조건부다.** [[키스톤-확정]]이 이미 권고한 옵션3(하이브리드: 캐피탈에 사후관리 도메인 편입)을 채택하는 순간, 동일한 분해 규칙이 최소 +1(자금용도외유용 점검 전문화 단위)을 요구해 **15**가 된다. 이는 숫자가 불안정하다는 뜻이 아니라 **"도메인팩 추가 시 증가"라는 원칙이 실제로 작동한다는 증거**다(§4).
- 병합 스트레스 테스트 5쌍 중 4쌍은 방어된다. 1쌍(준법 검토 vs 계약 체크리스트)은 근거가 가장 얇은 고리이며, 그 사실을 숨기지 않는다(§3).

## 1. 방법론 — "환원 불가 전문화 단위"의 정의

두 에이전트가 진짜로 달라지려면, 아래 4개 축 중 **최소 2개 이상**이 달라야 한다(1개만 다르면 이름만 바꾼 것일 위험이 크다 — §3에서 이 기준으로 스트레스 테스트한다).

| 축 | 의미 | 다르면 생기는 결과 |
|---|---|---|
| **데이터소스** | 이 판단이 무엇을 원료로 하는가 | 다른 데이터 파이프라인·다른 freshness 관리 |
| **실패양식** | 이게 틀리면 구체적으로 무엇이 깨지는가 | 다른 eval 기준·다른 사유코드 |
| **규제근거** | 어느 조문/가이드가 이 판단에 책임을 묻는가 | 다른 승인 레벨(L0~L4)·다른 설명요구권 대응 |
| **스킬셋** | 실제로 어떤 계산/추출/판단을 수행하는가 | 다른 모델 라우팅(로컬 PII vs 외부 LLM), 다른 프롬프트 |

D9의 경고를 스코프로 못 박는다: 멀티에이전트는 "전문화가 있을 때만" 정당화되고, 아니면 coordination overhead만 남는다(D9 "멀티에이전트는 '많은 에이전트'가 아니라 전문화 + 조정 비용을 감수할 가치가 있을 때만 쓰는 구조"). D8도 같은 방향이다 — AI 도구 추가가 자동으로 생산성을 높이지 않고 "업무-모델 적합성"에 달렸다(D8, 숙련 개발자 RCT에서 AI가 오히려 19% 느리게 만든 반례). 즉 **에이전트 수를 늘리는 것 자체가 목표가 아니라, 늘리지 않으면 설명력·검증력이 무너지는 지점에서만 늘린다**.

## 2. 바텀업 기능 분해

운영계약(`Case → AgentRun → Agent → Skill → Evidence → Approval → Audit`, canon §8)을 4개 축으로 전개한다: **오케스트레이션**(케이스 전체 흐름) × **여신·전세·피싱 3도메인**(확정 스코프, [[키스톤-확정]] 옵션3) × **통제기능**(준법·계약·포트폴리오, 도메인 횡단) × **고객·시스템 인터페이스**(사람/외부시스템 접점).

**합계**: 오케스트레이션 1 + 여신 3 + 전세 4(리드+전문가 3) + 피싱 1 + 통제 2(준법·계약) + 인터페이스 2(RM·은행연계) + 포트폴리오(도메인 횡단 통제) 1 = **14**.

두 가지가 이 표에서 눈에 띈다.

1. **"위험신호 조기감지"는 원래 여신 전용으로 세었는데도, canon의 직군×도메인 매핑([[agents-에이전트 아키텍처 재설계]])에서는 이미 사후관리 칸에도 걸쳐 있다.** 즉 지금의 14는 "3도메인만 본다"고 선언하면서도, 실제로는 사후관리라는 4번째 도메인의 그림자를 이미 일부 흡수하고 있다 — 이게 §4의 복선이다.
2. **"이상거래 탐지·차단"이라는 이름 자체가 이미 병합의 산물이다.** "탐지"(신호 판별)와 "차단"(집행)은 원칙적으로 서로 다른 실패양식(오탐 vs 과잉차단, D15가 둘을 별도로 다룸)을 갖지만, 캐논은 이 둘을 하나의 에이전트로 묶었다. 이는 팀이 이미 "환원 불가능하지 않으면 합친다"는 원칙을 실천해 왔다는 증거로 쓸 수 있다(§3에서 재사용).

## 3. 병합 스트레스 테스트

심사위원이 "이 둘 합쳐도 되잖아?"라고 물을 5개 쌍을 §1의 4축 기준으로 검증한다.

### 3-1. 전세가율 분석 vs 등기 권리 분석 vs 임차인 손실위험 — **방어**

| 항목 | 전세가율 | 등기 권리 | 임차인 손실위험 |
|---|---|---|---|
| 데이터소스 | 국토부 실거래가·공시가격 API | 등기부등본 열람 | 보증금·선순위채권·HUG 보증 데이터 |
| 실패양식 | 가격 괴리(공시 vs 시세) | 법적 권리 해석 오류 | 손실추정/보증범위 오판 |
| 산출물 | 가격 신뢰도 점수 | 권리관계 사실 확인서 | 손실 추정 + fallback 옵션 |

세 데이터소스가 서로 다른 기관·형식(공공데이터 API vs 등기부 열람 vs 보증기관 정보)이고, 실패양식도 "가격이 틀렸다"·"권리관계를 놓쳤다"·"손실액을 잘못 계산했다"로 완전히 다른 사유코드를 만든다. **가장 약한 고리는 등기 권리→임차인 손실위험 순서 의존성**(손실위험이 등기 권리의 출력을 입력으로 쓴다)이지만, 이는 병합 근거가 아니라 파이프라인 근거다 — D9가 권장하는 "오케스트레이터-워커, 전문가 분업형 handoff" 패턴과 정확히 일치하며, 각 단계가 별도 Evidence를 남겨야 설명요구권 대응이 가능하다(D14: "재산정 파이프라인"이어야 한다).

### 3-2. 위험신호 조기감지 vs 상환위험 분류 — **방어(중간 강도)**

| 항목 | 위험신호 조기감지 | 상환위험 분류 |
|---|---|---|
| 트리거 | 상시(신호 발생 시 자동) | 케이스 개설 후 촉발형 |
| 산출물 | Case 승격 여부 | 위험 밴드 + 사유코드 |
| 규제 의무 | 없음(아직 불이익 처분 전) | 신정법 §36조의2 설명요구권(불이익 처분에 걸림) |

"둘 다 리스크를 본다"는 점에서 병합 유혹이 가장 크다. 그러나 규제근거가 갈린다는 게 핵심 방어선이다 — 신용정보법 §36조의2는 "자동화평가의 결과"에 대해서만 설명요구권을 부여하므로, 아직 평가가 내려지지 않은 신호 단계(조기감지)와 이미 평가가 내려진 단계(분류)는 법적 의무의 발생 시점 자체가 다르다. 다만 이 구분이 "감시 vs 판단"이라는 다소 얇은 경계 위에 서 있다는 점은 인정한다 — 5쌍 중 두 번째로 약한 고리다.

### 3-3. 준법 검토 vs 계약 체크리스트 — **가장 얇은 고리, 정직하게 인정**

| 항목 | 준법 검토 | 계약 체크리스트 |
|---|---|---|
| 성격 | 법적 판단(신정법/PIPA 조문 적용, PII 반출 스캔) | 구조화 체크(문서 존재 여부, 특약 확인) |
| D9 분류 | Agent(분해·조정이 필요한 판단) | **Rule Engine(결정형 게이트)에 더 가까움** |
| 승인 레벨 | L2~L4 | L1~L3 |

이 쌍은 두 문서(`agents-에이전트 아키텍처 재설계`, `agents-v2-paperclip기반-재설계`) 모두 정면으로 다루지 않았던 지점이다. D9는 명시적으로 "결정형 게이트는 규칙 계층, 설명·추출·추천·요약은 모델 계층"으로 분업하라고 권고한다. 계약 체크리스트("이 서류가 있는가/없는가")는 이 정의상 Rule Engine 영역에 더 가깝고, 준법 검토("이 행동이 신정법/PIPA를 위반하는가")만 진짜 판단형 Agent다. **정직한 평가**: 현재 구조에서 둘을 분리 유지하는 근거는 (a) 서로 다른 데이터소스(문서 OCR vs 법령 RAG)와 (b) 서로 다른 실패 회복성(서류 누락은 재요청으로 복구되는 운영 리스크, PII 반출은 복구 불가능한 규제 리스크)이지 강한 환원불가성은 아니다. 리소스가 빠듯해지면 계약 체크리스트를 준법 검토 산하 스킬(compliance check의 하위 skill)로 흡수하는 것도 방어 가능한 대안이며, 그 경우 **13**이 된다 — 이 문서는 이 가능성을 숨기지 않는다.

### 3-4. RM 보좌 vs 은행 연계 — **방어(가장 강함)**

| 항목 | RM 보좌 | 은행 연계 |
|---|---|---|
| 소비자 | 고객(대면 커뮤니케이션) | 내부 시스템(API/계열사 연동) |
| 실패양식 | 안내 오류, 어조 부적절(회복 가능) | 회사 경계 침범, 접속기록 누락(회복 불가능한 기술적 사고) |
| 규제근거 | "AI는 보조수단" 원칙(D14) | 전자금융거래법 §22, 신용정보업감독규정 접속기록 보존(D15) |
| 승인 레벨 | L1~L2 | L2~L4 |

소비자(고객 vs 시스템)·실패양식(커뮤니케이션 품질 vs 보안/경계 침범)·규제근거·승인 레벨 4축 모두 다르다. 5쌍 중 가장 명확하게 방어된다.

### 3-5. 포트폴리오 분석 vs 위험신호 조기감지 — **방어**

| 항목 | 포트폴리오 분석 | 위험신호 조기감지 |
|---|---|---|
| 분석 단위 | 지점/계열사 집계(다수 케이스) | 개별 고객 케이스 1건 |
| 실패양식 | 공정성 격차 은폐(승인율/TPR 격차, D14) | 개별 신호 누락(thin-file) |
| 소비자 | 지점장/경영진(KPI 대시보드) | RM/사후관리 담당자(케이스 단위 트리아지) |

분석 단위(집계 vs 개별)와 소비자가 다르다. BCBS239가 요구하는 데이터 품질 축(집계 리포트 정확성)과 개별 신호 freshness는 서로 다른 실패 감시 체계다.

## 4. 정직한 조정 — 사후관리 편입 시 +1

[[키스톤-확정]]은 옵션3(하이브리드: 은행=여신·전세보호·피싱 / 캐피탈=여신·**사후관리**)을 권고안으로 제시했고, 그 근거는 2026-06-30 시행된 은행연합회 "자금용도외 유용 사후점검준칙" 개정이다 — 개인사업자 점검 의무 대상이 1억→5,000만원, 법인이 5억→3억으로 하향되고 법인 부동산 임대사업자 시설자금까지 확대됐다(D16, [[키스톤-역할축-검증]] §1 재인용). 이 규제는 **자금 집행/용도증빙 문서를 대조해 용도외유용을 적발**하는 작업을 요구하는데, 이 실패양식(용도외유용 적발 실패)과 규제근거(2026-06-30 신규 준칙)와 데이터소스(집행/용도증빙 문서)는 현재 14개 어느 에이전트의 명시된 스킬에도 없다 — 위험신호 조기감지는 "매출·상권 신호"를, 은행 연계는 "시스템 handoff"를, 계약 체크리스트는 "계약 체결 시점 서류"를 다룰 뿐, "이미 실행된 여신의 사후 용도 대조"는 다른 작업이다.

**따라서 사후관리를 4번째 도메인으로 정식 편입하면, 같은 4축 기준을 일관되게 적용할 때 최소 +1(자금용도외유용 점검 전문화 단위)이 필요하다 — 15.** 이것이 §0에서 예고한 조건부 결론이다. 이는 숫자의 불안정성이 아니라, `agents-v2-paperclip기반-재설계` 문서가 이미 제시한 원칙("새 계열사/도메인은 전체 로스터를 복제하지 않는다. 도메인팩으로 추가한다")이 실제로 작동하는 사례다 — 사후관리 도메인팩 = Lead 없이 Specialist 1(자금용도외유용 점검) + 기존 준법/포트폴리오/은행연계 재사용으로 구성 가능하다.

## 5. 원칙 한 문장

> **14는 우리가 고른 목표가 아니라, 확정 스코프(여신·전세·피싱 3도메인 + 오케스트레이션 + 통제기능)에서 {데이터소스 × 실패양식 × 규제근거 × 스킬}이 환원 불가능하게 다른 전문화 단위의 수다. 도메인을 추가하면(예: 사후관리 편입 시 +1) 증가하고, 스트레스 테스트에서 환원 가능성이 확인되면(예: 준법 검토·계약 체크리스트 통합 시 -1) 감소한다 — 14는 스냅샷이지 신앙이 아니다.**

## 6. 심사 Q&A 예상질문

**Q1. 왜 5개 직군(RM·여신심사·사후관리·준법·AML)이 아니라 14개 에이전트인가?**
5개는 조직/사이드바 축(누가 로그인해서 보는가)이고, 14개는 실행 단위(무엇이 판단하는가)다. 하나의 "여신심사" 직군 안에도 위험신호·상환위험·정책매칭처럼 데이터소스와 실패양식이 다른 작업이 섞여 있어, 하나의 에이전트로 뭉치면 사유코드가 뭉개지고 설명요구권 대응이 약해진다(D14, [[키스톤-역할축-검증]]).

**Q2. 20개, 30개로 더 늘리지 않는 이유는?**
D9가 명시적으로 경고한다 — 멀티에이전트는 전문화가 있을 때만 정당화되고, 그 외에는 coordination overhead·latency·failure mode만 늘린다. 본선 MVP 범위에서 활동 로그·eval·UI 구현 부담도 동반 증가한다. 신규 도메인은 core roster 증식이 아니라 도메인팩(Lead + Specialist 1~3)으로 붙인다.

**Q3. 사후관리를 넣으면 14가 아니라 15 아닌가?**
정직하게 그렇다. 2026-06-30 시행 규제(자금용도외유용 사후점검준칙)를 스코프에 넣으면 기존 14개 중 어느 것도 커버하지 못하는 새 전문화 단위가 1개 필요해 15가 된다. 이게 결함이 아니라 "도메인팩" 설계 원칙이 실제로 검증 가능하다는 증거다.

**Q4. 전세가율·등기 권리·임차인 손실위험, 그냥 "전세 에이전트" 하나면 안 되나?**
데이터소스(실거래가 API vs 등기부 열람 vs 보증기관 정보)와 실패양식(가격 괴리 vs 법적 권리 오류 vs 손실추정 오류)이 다르다. 국토부 스스로 실거래가 시스템 수기입력 오류를 인정했고(D14), 등기 권리와 보증금 손실 판단이 뒤섞이면 어느 단계에서 틀렸는지 사후 추적이 불가능해진다.

**Q5. 준법 검토와 계약 체크리스트, 사실 하나로 합쳐도 되지 않나?**
이 지점은 가장 약한 고리임을 인정한다. 계약 체크리스트는 D9 기준 결정형 Rule Engine 영역에 가깝고, 준법 검토만 진짜 판단형 Agent다. 지금은 데이터소스(문서 OCR vs 법령 RAG)와 실패 회복성(서류 누락은 복구 가능, PII 반출은 복구 불가능) 차이로 분리 유지하지만, 리소스가 빠듯해지면 계약 체크리스트를 준법 검토의 하위 스킬로 흡수하는 것도 방어 가능한 대안이다(그 경우 13).
