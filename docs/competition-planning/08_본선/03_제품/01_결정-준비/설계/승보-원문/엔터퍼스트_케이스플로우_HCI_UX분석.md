# Enter-First Case Flow UI HCI/UX 분석

## 1. 컨셉 정의

**Enter-First Case Flow UI**는 금융 업무 담당자가 마우스로 화면을 탐색하며 처리하는 방식이 아니라, AI가 정리한 케이스 흐름을 눈으로 훑고 `Enter`로 다음 단계에 넘기는 업무 처리 UI다.

핵심은 키보드 단축키 자체가 아니다. 사용자의 감각과 주의를 불필요한 탐색, 조작, 메뉴 선택에 쓰지 않게 하고, **업무상 필요한 확인과 최종 판단에만 남기는 것**이 핵심이다.

기본 흐름:

```text
케이스 확인 -> 핵심 근거 훑기 -> Enter -> 다음 단계/다음 역할로 전달
```

예외 흐름:

```text
케이스 확인 -> 위험/중요 신호 감지 -> Enter Enter -> 중요 표시/상위 검토/감사 플래그
```

최종 책임 흐름:

```text
최종 승인 필요 -> 근거 확인 -> 마우스 클릭으로 승인
```

## 2. UX 핵심 명제

이 UI는 사용자를 화면의 탐색자로 만들지 않는다.

사용자를 **검토자, 통과자, 예외 감지자**로 만든다.

기존 업무 UI에서는 사용자가 매번 다음 질문을 직접 해결해야 한다.

- 어디를 봐야 하는가?
- 어떤 메뉴를 눌러야 하는가?
- 이 케이스의 핵심은 무엇인가?
- 다음 담당자는 누구인가?
- 이 판단을 승인해도 되는가?

Enter-First UI에서는 시스템과 AI가 먼저 업무 흐름을 정리한다.

```text
볼 것 정리
-> 기본 액션 제안
-> Enter로 진행
-> 예외만 깊게 보기
-> 최종 승인만 명시 클릭
```

따라서 사용자의 감각은 탐색, 판단, 조작 전체에 분산되지 않고 **확인과 예외 감지**에 집중된다.

## 3. HCI 관점 분석

### 3.1 인지 부하 감소

업무자는 판단 자체보다 “무엇을 봐야 하는지 찾는 비용”에 많이 지친다. 여러 시스템, 문서, 탭, 메뉴를 오가면 작업자는 매 케이스마다 화면의 의미를 다시 해석해야 한다.

Enter-First UI는 케이스마다 다음 정보를 같은 위치와 같은 구조로 반복 제공한다.

- 한 줄 요약
- 위험도
- 추천 액션
- 핵심 근거
- 다음 역할

이는 HCI의 `recognition over recall` 원칙과 맞다. 사용자가 정보를 기억해서 찾아가는 것이 아니라, 화면이 필요한 판단 재료를 보여주고 사용자는 인식하고 확인한다.

### 3.2 운동 부하 감소

마우스 중심 업무는 생각보다 피로도가 높다.

- 커서 이동
- 버튼 탐색
- 클릭 타겟 조준
- 화면과 손의 반복 전환
- 메뉴/탭/필터 조작

Enter 중심 UI는 손의 움직임을 고정한다.

```text
눈: 핵심 정보 훑기
손: Enter
예외: Enter Enter
최종 책임: 마우스 클릭
```

하루에 수십~수백 건의 케이스를 처리하는 금융 업무에서는 이 차이가 누적된다. 사용자는 화면 조작보다 케이스 판단에 더 많은 에너지를 남길 수 있다.

### 3.3 Hick's Law: 선택지 감소

Hick's Law에 따르면 선택지가 많을수록 결정 시간이 늘어난다. 기존 대시보드형 UI는 버튼, 메뉴, 탭, 필터, 상세 링크가 많아 사용자가 매 순간 선택해야 한다.

Enter-First UI는 대부분의 순간에 기본 액션을 하나로 둔다.

- `Enter`: 기본 추천대로 진행
- `Enter Enter`: 중요/예외/재검토 표시
- `Mouse Click`: 책임 있는 최종 승인

이 구조는 반복 업무에서 결정 피로를 낮춘다. 사용자는 매번 “무슨 버튼을 눌러야 하지?”를 고민하지 않고, 추천된 흐름을 검토한 뒤 통과시킨다.

### 3.4 Fitts's Law: 클릭 부담 최소화

Fitts's Law 관점에서 클릭은 타겟까지 이동하고 정확히 조준해야 하는 동작이다. 자주 반복되는 액션일수록 마우스 클릭은 피로와 시간을 만든다.

Enter-First UI는 자주 반복되는 진행 액션을 키보드로 옮긴다. 반대로 책임이 큰 최종 승인만 마우스 클릭으로 남긴다.

이 분리는 중요하다.

모든 작업을 Enter로 처리하면 습관적 통과가 생길 수 있다. 최종 승인은 일부러 마찰이 있어야 한다. 따라서 **낮은 책임의 흐름 진행은 Enter, 높은 책임의 최종 승인은 클릭**으로 나누는 것은 효율성과 안전성을 동시에 확보하는 설계다.

### 3.5 Progressive Disclosure

Enter-First UI는 처음부터 모든 정보를 보여주지 않는다. 기본 화면에서는 판단에 필요한 최소 정보만 제공한다.

- 요약
- 위험도
- 추천 액션
- 근거 3개
- 다음 역할

상세 근거, 원문 문서, 감사 로그, PII 접근 내역은 필요할 때만 연다. 이는 정보를 숨기는 것이 아니라, **판단 순서에 맞게 늦게 보여주는 것**이다.

업무자는 대부분의 케이스를 빠르게 훑고, 예외 케이스에서만 깊게 들어간다.

### 3.6 Flow 상태 유지

업무자는 케이스를 처리할 때 흐름이 끊기면 피로해진다. 탭 이동, 팝업, 검색, 메뉴 탐색이 많으면 리듬이 깨진다.

Enter-First UI는 업무를 리듬화한다.

```text
보고
확인하고
넘기고
다음 케이스
```

이 흐름은 심사, 모니터링, 이상거래 탐지, 고객센터 후처리, 내부 승인 업무처럼 대량 케이스를 처리하는 환경에 잘 맞는다.

## 4. 금융 업무에 적합한 이유

금융 업무는 완전 자동화가 어렵다. 책임, 감사, 규정, 개인정보, 설명 가능성이 필요하기 때문이다.

반대로 모든 판단을 사람이 직접 찾고 처리하는 방식도 비효율적이다. 금융 업무의 병목은 종종 데이터 부족이 아니라, **분산된 근거를 모으고 다음 역할로 넘기는 과정**에서 생긴다.

Enter-First UI는 AI와 사람의 역할을 다음처럼 나눈다.

| 주체 | 역할 |
|---|---|
| AI | 정리, 추천, 근거 수집, 다음 역할 제안 |
| 사람 | 확인, 예외 감지, 최종 승인 |
| 시스템 | 로그, 감사, PII 통제, 역할 간 전달 |

즉 AI가 판단을 대체하는 것이 아니라, 사람이 판단할 수 있도록 감각 비용을 줄인다.

## 5. 차별성 평가

### 5.1 챗봇형 AI와의 차별성

많은 AI 업무 도구는 채팅창을 붙이는 방식으로 출발한다. 그러나 채팅형 UI는 사용자가 질문을 잘해야 한다. 업무 처리 상황에서는 질문을 만드는 시간 자체가 비용이 된다.

Enter-First UI는 질문형이 아니라 **케이스 흐름형**이다.

AI가 먼저 정리하고, 사용자는 훑고 넘긴다. 따라서 대량 업무 처리에는 챗봇보다 더 빠르고 일관된 경험을 제공할 수 있다.

### 5.2 대시보드형 AI와의 차별성

대시보드는 현황을 보는 데 강하지만 처리에는 약하다. 지표와 그래프는 많이 보여주지만 “그래서 다음에 뭘 해야 하는가?”가 약할 수 있다.

Enter-First UI는 케이스마다 다음 액션이 있다.

즉 보는 화면이 아니라 **처리하는 화면**이다.

### 5.3 RPA/자동화와의 차별성

RPA는 사람이 하던 클릭을 자동화한다. 그러나 업무 판단 구조 자체를 바꾸지는 못한다.

Enter-First UI는 클릭 자동화가 아니라 **판단 흐름 재설계**다. 역할별 게이트를 따라 케이스가 이동하고, 사람은 자신의 역할에서 확인과 예외 감지를 수행한다.

### 5.4 일반 워크플로 툴과의 차별성

일반 워크플로 툴은 상태 전이와 담당자 배정에 집중한다. Enter-First UI는 그 위에 AI 근거, 요약, 승인, 감사, PII 통제를 결합한다.

따라서 단순 티켓 시스템이 아니라 **AI-assisted decision gate**라고 볼 수 있다.

## 6. UX 리스크와 방어 설계

Enter-First UI는 빠른 만큼 위험도 있다.

| 리스크 | 설명 | 방어 설계 |
|---|---|---|
| 습관적 Enter | 사용자가 중요한 케이스를 무심코 넘길 수 있음 | 고위험 케이스는 Enter만으로 진행 불가 |
| AI 과신 | AI 추천이 너무 강하게 보이면 사람이 검토를 생략할 수 있음 | 추천과 근거를 분리 표시, 확신도/불확실성 표시 |
| Enter Enter 모호성 | 더블 Enter의 의미가 불명확하면 실수 가능 | `중요 표시/상위 검토` 등 의미를 고정 |
| 최종 승인 남발 | 최종 승인 클릭이 너무 자주 나오면 흐름이 깨짐 | 승인 클릭은 책임이 큰 마지막 단계에만 사용 |
| 빠른 오판 | 근거가 부족한데 UI만 빠르면 오판도 빨라짐 | 근거 부족 케이스는 자동으로 보류/재검토 큐로 이동 |

핵심 규칙:

```text
Enter = 낮은 책임의 진행
Enter Enter = 예외/중요 표시
Mouse Click = 책임 있는 최종 승인
```

## 7. 디자인 원칙

1. **눈 이동을 최소화한다.**
   - 핵심 요약, 위험도, 추천 액션, 다음 역할은 항상 같은 위치에 둔다.

2. **마우스 조작을 최종 책임에만 남긴다.**
   - 반복 진행은 Enter, 최종 승인은 클릭으로 분리한다.

3. **역할과 도메인을 섞지 않는다.**
   - 사용자는 역할로 들어오고, 역할 안에서 케이스를 처리한다.

4. **AI 추천은 결론이 아니라 검토 대상이다.**
   - 추천, 근거, 불확실성, 담당자 액션을 함께 보여준다.

5. **예외는 흐름 위로 떠오르게 한다.**
   - 사용자가 찾는 것이 아니라, 고위험/중요 케이스가 자동으로 드러나야 한다.

6. **감사 가능성을 기본 구조에 포함한다.**
   - AI 추천, 사람 확인, 최종 승인, 사용 데이터가 모두 로그로 남아야 한다.

## 8. PM 검토용 결론

Enter-First Case Flow UI의 차별성은 키보드 조작이 아니라, **금융 업무의 판단 흐름을 역할 기반 게이트로 재설계했다는 점**이다.

이 UI는 AI 챗봇도 아니고, 단순 대시보드도 아니며, RPA도 아니다.

AI가 케이스를 정리하고, 사람은 눈으로 훑고, Enter로 흐름을 통과시키며, 최종 책임만 명시적으로 클릭한다.

한 문장으로 정리하면 다음과 같다.

> AI가 업무 감각을 미리 정리하고, 담당자는 눈으로 훑고 Enter로 케이스를 통과시키며, 최종 책임만 클릭으로 남기는 역할 기반 금융 업무 콘솔.
