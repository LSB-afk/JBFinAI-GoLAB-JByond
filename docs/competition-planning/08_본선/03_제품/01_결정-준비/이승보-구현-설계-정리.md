## 만든 것(구현 현황)

- **계열사 분류 체계**: JB금융그룹을 은행 계열사(전북은행·광주은행)와 비은행 계열사(JB우리캐피탈)로 나눠 접근하는 구성을 직접 만들었다.
  > "제가 방금 얘기했던 것처럼 JB금융그룹의 큰 상위 개념에서 하위 개념으로 내려왔을 때, 은행 계열사와 비은행 계열을 나눠봤습니다. 은행 계열사는 전북은행과 광주은행, 비은행 계열사는 JB우리캐피탈로 일단 나눠봤고요." (7/1, 이승보)

- **은행권 하네스(대시보드) 및 담당자별 역할 대시보드**: UX/UI 초안을 다듬고 실제로 "구동 동작"이 되게 만드는 작업을 진행 중이었다.
  > "훨씬 더 깔끔해졌습니다. 저는 민주님께서 작업을 잘하셨고, 그래서 이런 흐름을 계속 제가 다듬고 있었고, 잘 구동 동작이 되게 하고 있었습니다. 그리고 그거랑 같이 우리 비은행권 하네스도 동작업을 하고 있었어요." (7/2, 이승보)

- **JB우리캐피탈(비은행권) 하네스**: JB캐피탈 사이트를 분석해 캐피탈 쪽 화면/흐름을 별도로 만들고 있었다(일정 때문에 잠시 중단).
  > "JB 캐피탈 사이트에 들어가서 분석하면서, 여기 보니까 이렇게 나와 있더라고요. 어떤 게 있는지 흐름을 계속 분석하면서 같이 작업하다가 일정이 있어서 잠깐 멈췄고..." (7/2, 이승보)

- **역할별 대시보드 다듬기**: "역할도 제가 작업해서 다 다듬으셨어요. 역할별로 다듬어 주셨고, 이 정도입니다. 이거 오늘 내일 중으로 더 다듬을 겁니다." (7/2, 이승보)

- **도메인 4개 + 에이전트 로스터(14개) — 김주용이 승보 작업물로 지칭**: 여신·여신 사후관리·보이스피싱·전세사기 4개 도메인과 14개 에이전트 이름 목록이 이미 만들어져 있음을 김주용이 확인·소개.
  > "그러면 보면 계열사가 전북, 광주, 캐피탈 OK, 일단 승모님 하신 게 만들어져 있고, 도메인 네 개 주요 업무를 어떤 걸 하냐, 여신, 여신 사후 관리, 보이스피싱, 전세 사기 관련한 것도 작업을 쭉 하고 있고, 로스터가 에이전트 말하는 건데 14개의 이름..." (7/2, 김주용)

- **은행 콘솔(전북은행 기준) + 캐피탈 콘솔 병행 개발**: 7/3 시점, 그룹 콘솔·전북은행 콘솔·광주은행 콘솔·캐피탈 콘솔을 각각 만드는 대신, 그룹 콘솔이 은행 콘솔과 구조상 거의 같다고 판단해 은행(전북은행)과 캐피탈 두 축으로 좁혀 만들고 있다고 스스로 정리.
  > "이 콘솔이 금융 그룹 콘솔이랑 전북은행, 광주은행, 우리은행, 캐피탈 콘솔. 이제 세 개를 더 만들어야 되는데. 너무 많은 것 같아서, 그런데 사실 이 금융 그룹 콘솔이 지금 현재 있는 콘솔이 어떻게 보면 은행 콘솔이랑 거의 비슷하거든요. ... 이거 지금 현재 메인 화면을. 전북은행으로 하나를 전북은행으로 넣고 사실 전부 은행으로 넣고, 그리고 현재 제가 또 만들고 있는 우리 캐피탈을 이렇게 캐피탈을 만들고 있는 것을 더 이제 디벨롭 시켜서. 이렇게 두 가지만 가져가는 게 어떠신가요?" (7/3, 이승보)

- **프롬프트 고도화 실험**: 정리 화면을 캡처해 내용 정리·고도화를 AI에게 요청하는 작업을 시도.
  > "그래서 바로 이거 프롬프트 고도화 한 번 해봤습니다." / "제가 그냥 한 번 이거 지금 돌 이거 캡처해서 이 내용도 정리하고 고도화해 달라고 한 번 해보고 있습니다." (7/3, 이승보)

- **예선(예선전) 산출물과의 차이**: 예선 때는 역할별로 나뉘지 않은 단일 메인 대시보드만 있었다는 것을 본인이 확인 — 즉 역할별 대시보드/토글 구조는 본선을 위해 새로 설계·구현한 것.
  > "아니요, 예선에 없었습니다. 이렇게 따로 역할이 나뉘어져 있어요." / "그냥 딱 이 대시보드, 메인 대시보드만 있었어요. 예전 나올 때." (7/3, 이승보)

## 설계 사고(왜 이렇게)

- **자동화를 넘은 "의사결정까지" 하는 RM 역할이 대회 니즈라는 해석**에서 출발해 도메인을 나눴다.
  > "얘네가 원하는 것은 AI 에이전트를 활용해서 자동화를 넘어서 의사결정까지 구축해주는 RM의 역할인 것 같습니다. 여신 담당자 역할까지 원하는 것 같고, 그래서 보이스피싱, 여신 사후관리, 전세 사기 관련까지 원하는 것 같더라고요." (7/1, 이승보)

- **AI에게 프롬프트 고도화를 맡겼다가 내용이 통제 불가하게 바뀌자, 직접 수동 개발로 되돌아간 선택**.
  > "이거 작업을 한 번 쭉 돌렸었는데, 이 내용으로 디스코드 내용을 활용해서 프롬프트를 고도화시켜 작업을 돌렸더니 저희가 기존에 만들었던 것에서 내용이 갑자기 말도 안 되게 싹 다 바뀌어버렸어요. 그래서 너무 놀라서 내용을 다시 지우고, 일단 제가 하나하나 다시 개발을 시작했거든요." (7/1, 이승보)

- **은행 계열사 1곳 + 비은행 계열사 1곳으로 트랙을 좁힌 이유**: 전부 다루면 양이 지나치게 많아진다는 판단.
  > "그래서 제가 생각한 것은 은행 계열사 하나랑 비은행 계열사 하나를 넣어서 이 두 가지를 타겟으로 잡고 개발해보면 어떨까 하는 거예요. ... 필요한 것만 딱딱 넣어서요. 사실 이것만 해도 양이 엄청 많을 것 같아요." (7/1, 이승보)

- **자유주제/지정주제를 분리하지 않고 설계에 복합적으로 포함시킨 선택** — 최영욱이 "자유 주제인 건가요?"라고 묻자:
  > "저는 이거 설계할 때 그거를 다 복합적으로 포함시켜버렸어요." (7/2, 이승보)

- **역할별 다양성(RM·기업 여신 담당자·전세 보호 담당자 등)을 담당자 대시보드별로 구현하는 것이 원래 설계 목적**이었다고 스스로 정리(김주용의 "포인트가 정리 안 된 느낌" 지적에 대한 답).
  > "원래 제가 설계한 목적은 자유 주제로 이런 다양한 역할, RM, 기업 여신 담당자, 전세 보호 담당자 이분들이 주용님께서 말씀하신 기능들을 담당자 대시보드별로 각각의 역할을 해내는 거였어요." (7/2, 이승보)

- **핵심 도메인을 좁혀 차별점을 살리는 방향**: RM·보이스피싱·전세보호(+기업여신)를 우선순위로 제안했고, 이후 발표용으로는 RM·전세사기·보이스피싱 세 가지로 더 좁히자고 제안.
  > "일단 제가 생각하기에는. 어떤 것을 가져가면 차별점이 좋을까 해서. 이 도메인이 일단 RM 하나랑 보이스피싱이랑 전세보호. 이거 세 가지는 꼭 가져가면 일단 좋을 것 같아요. 그리고 기업 여신까지." (7/3, 이승보)
  > "제가 핵심적인 것을 주영 님이랑 말했던 게, 일단은 저희 계열사가 은행이랑 캐피탈 두 개로 줄었고, 도메인 역할, RM, 기업 여신, 전세 보호, 보이스피싱, 내부 통제, 준법 감시, 일단 이 4개에서 5가지. 제가 봤을 때는. 이 RM이랑. 전세, 전세사기, 그리고 보이스피싱 이 세 가지만 얘기를 해주면 어떨까?" (7/3, 이승보)

- **완성도 시연이 점수에 유리하다는 판단으로 실제 로컬 모델 구동 연결을 제안**.
  > "실제 저희 올라마 같은. 주영 님께서 말씀하신 거 이런 거 진짜 연결해서. 우리는 발표할 때나 발표 자료 넣거나 이렇게까지 연결을 해서 해봤다, 실제 이렇게 구동까지 된다는 내용을 넣으면 무조건 좋을 것 같아요." / "완성도 측면에서 점수 엄청 많이 받을 것 같아요." (7/3, 이승보)

- **자체 평가: 다른 상위권 팀들과 비교해 기획·설계가 앞선다고 판단**, 페이퍼클립을 설계 기반으로 삼았음을 언급.
  > "다른 분들이 올린 것을 보았을 때 너무 다들 뻔하고, 저희같이 설계한 것들은 그런 기획과 설계가 부족한 게 많았어요. 저는 압도적인 느낌을 받았어요. 사실 라이브 코딩 때 페이퍼 클립 기반으로 이거 설계를 한 거잖아요. 1등 한다고는 못하겠는데, 좋은 성적은 있을 것 같아요." (7/2, 이승보)

## 역할 토글·화면 구성(프로토타입)

사용자가 특히 확인을 요청한 부분. 7/2 회의에서 이승보가 직접 설명한 원문 그대로:

- **최초 설계안(폐기)**: 역할별로 완전히 분리된 로그인 화면 → 역할별 별도 메인 화면.
  > "원래 제 생각은 이거를 배포했을 때 로그인 화면에서, 처음 메인 화면에서 각각의 롤에 맞는 아이디를 생성시켜 줘서, 예를 들어 RM 담당하시는 분의 아이디를 만들어서 로그인을 하면 RM 대시보드만 뜨게 하는 생각이었어요." (7/2, 이승보)

- **폐기 이유**: 각각 따로 메인 화면으로 만드니 "너무 밋밋"하고, 팀의 의도(다양한 역할·기능을 한 시스템 안에서 보여주려는 의도)가 심사위원/보는 사람에게 와닿지 않을 것 같다는 판단.
  > "그런데 우리의 의도를 보여주고 싶어서 원래는 처음에 그렇게 설계를 해봤거든요. 각각 따로따로 메인 화면으로 설계를 해봤는데 너무 밋밋한 거예요. 그리고 보여주는데 이분들이 확 와닿지 않을 것 같다는 생각 때문에..." (7/2, 이승보)

- **채택한 구조**: 왼쪽 사이드바에 담당자(역할) 목록을 빼두고, 하나를 누르면 그 담당자의 대시보드로 전환되는 방식. "오늘 처리할 AI 자동화 관리" 등 왼쪽의 다른 대시보드 항목도 각 담당에 맞게 전부 바꾸는 것이 개발 설계 목표.
  > "일단은 왼쪽 사이드바에 빼서 눌렀을 때 담당자별로 본인들의 대시보드가 나오면서 사용할 수 있게 했습니다. 여기 오늘 처리할 AI 자동화 관리, 왼쪽에 있는 다른 대시보드도 각각 담당에 맞는 대시보드로 싹 맞게 바꾸려는 게 제 개발 설계였습니다." (7/2, 이승보)

- **참고: "토글"이라는 표현 자체는 정확한 명칭이 아니라는 지적이 있었음(7/1, 김민주)** — 즉 사용자가 말한 "토글"은 팀 내부에서도 정식 용어로 확정된 것은 아니고, 실제로는 사이드바 클릭 전환 방식으로 구현·설명됨(원문 불명확: "토글"이라는 단어를 이승보 본인이 썼는지는 확인 안 됨).
  > "이거 와이어프레임도 다시 짜서 해야 해요. 이런 토글이나 이런 게 사실 맞는 표현법은 아니어서 다요." (7/1, 김민주)

- **배포/로그인 흐름에 대한 후속 설명(김주용의 질문에 답하는 형태로 재확인)**: 로그인 시 부여된 코드로 본인 역할 대시보드에 접속하고, 실제 업무 실행은 에이전트(로컬 모델 또는 Claude/GPT 등)가 수행하며, 결과를 검토·수정·승인하는 흐름.
  > "주로 RM이 하는 역할들을 심어놓고, 은행에서도 자동화할 수 있는 다양한 역할들을 심어놓고, 담당자가 여기에 로그인해서 접속해서 실제 업무를 주고 입력 데이터를 넣고, ... 결과 화면이 나오겠죠. ... 최종 승인, 최종 도장이라고 할까, 최종 확인을 하고 종료가 되겠죠." (7/2, 김주용 — 이승보가 보여준 설계를 김주용이 재서술·확인)
  > "일단 확정된 것은 저희 아까 얘기했던 담당자별 AI 협업 콘솔, 대시보드, 하네스라고 할 수도 있고 결국 로그인해서 코드를 넣어서 본인 역할 대시보드를 볼 수 있다." (7/3, 김주용)

## 운영자 어드민 중심 개발 방향 / 실사용자 로그인 구조

- 이승보 본인이 명시적으로 "운영자 어드민 중심"이라는 표현을 쓴 원문은 확인되지 않음(원문 불명확). 다만 그가 설명한 로그인 구조는: 배포 시 로그인 화면에서 **역할별 아이디를 생성**하고, 각 아이디로 로그인하면 해당 역할(RM 등)의 대시보드만 노출되는 구조였다(최초 설계) → 이후 사이드바 토글형으로 변경.
  > "예를 들어 RM 담당하시는 분의 아이디를 만들어서 로그인을 하면 RM 대시보드만 뜨게 하는 생각이었어요." (7/2, 이승보)

- 김주용이 이 구조를 두고 "개인 로그인"·"개인 코드"가 필요할 것이라 재확인 질문했고, 이승보가 "맞아요, 딱 그 부분이에요"로 긍정.
  > "얘는 개인 코드도 필요할 텐데, 아마 개인 로그인이든 그런 거겠죠. 한 명이 이 화면을 보고 자신의 역할에 맞는 업무를 쓰겠죠. 그러면 저 역할이라는 것을 저희가 볼 수 있게 해놓은 거죠, 그냥." (7/2, 김주용) → "맞아요. 딱 그 부분이에요." (7/2, 이승보)

## 기술 선택

- **모델 구동 방식 — 하이브리드(Claude/Codex 연결 + 로컬 모델)**: 이 항목은 원문상 발화 주체가 **김주용**이며("제가 저번에 말씀드린 거는"), 이승보가 직접 하이브리드를 이 표현으로 제안한 문장은 확인되지 않음(원문 불명확 — 다만 로컬 모델·보안 필요성 자체는 7/1 회의에서 김주용이 먼저 제기했고 이승보는 별도로 명시적 반대를 하지 않음).
  > "그다음에 디포 구동 모델, 그러니까 어떤 모델 쓸 거냐? 배포형이냐, 로컬이냐, 하이브리드냐... 그래서 제가 저번에 말씀드린 거는. 일단 클로즈나 코덱스 연결할 수 있게는 해두고. 그리고 로컬에서도 올라마나 다른 것을 활용해서 실제 모델, 로컬 모델을 실동작으로 시연하는 것." (7/3, 김주용)

- **Claude API — 이승보가 직접 다룬 부분**: 7/2 시점에는 아직 API를 호출하지 않았고("잘못하면 위험할 것 같아서 다 같이 있을 때 하려고"), 망상궤도 공용 계정에 Claude 아이디가 있는 것만 확인. 7/3 시점에는 "클로드 API까지 있어서 좀 수월하게 만들 수 있을 것"이라고 언급 — API 확보 이후 개발 용이성에 대한 기대.
  > "저는 이거 API 아직 호출 안 했거든요. 해보려고 하니까 뭔가 잘못하면 위험할 것 같아서 다 같이 있을 때 하려고 아직 안 했습니다." (7/2, 이승보)
  > "망상궤도 아이디 들어가 보니까 클로드 아이디가 있더라고요." / "아직 API 넣지 않았어요. 망상궤도 클로드 아이디 있는 것만 확인했어요." (7/2, 이승보)
  > "사실 이거 이제 제 생각에는 만드는 거는 이거 클로드 API까지 있어서 좀 수월하게 만들 수 있을 거라고 생각해요, 저는." (7/3, 이승보)

- **로컬 모델(Ollama) 연결 — 이승보의 제안**: 실제로 Ollama 등을 연결해 로컬 모델이 구동되는 것까지 보여주면 완성도 점수에 유리하다고 제안(제안 시점 기준 미연결, 아래 "실동작 vs 미완" 참고).
  > "실제 저희 올라마 같은. ... 진짜 연결해서. ... 실제 이렇게 구동까지 된다는 내용을 넣으면 무조건 좋을 것 같아요." (7/3, 이승보)

- **DB 연동 방식(정적/서버/하이브리드) — 결정 보류, 승보와 논의 예정으로만 언급**: 발화 주체는 김주용이며, "저랑 승부님(승보님)이랑 얘기를 해 보겠습니다"로 위임. 이승보 본인이 정적 MVP를 명시적으로 선택했다는 원문은 확인되지 않음(원문 불명확).
  > "다섯 번째. DB 연동. 이게 기업의 은행 시스템에 DB가 있을 텐데, 거기에 연결하고 정적으로 할지, 아니면 실제 서버 만들어서 할지 하이브리드로 할지, 이거는 사실 저랑 승부님이랑 얘기를 해 보겠습니다." (7/3, 김주용)

- **에이전트 무기(스킬·MCP·은행 DB·국가 오픈 API·LLM)**: 7/3 회의에서 "확정된 것"으로 정리된 항목이며, 발화 주체는 김주용. 이승보가 이 표현을 직접 썼는지는 확인되지 않으나, 관련 개념(스킬·MCP·공공 API 연결)은 7/2에 이승보도 "무기 에이전트가 무기가 있어야 되는데, 그중에 하나가 이런 공공 API를 가져다 올 수 있는 거다"로 동의·부연.
  > "두 번째, 에이전트가 주로 뭘 하냐, 어떤 무기들이 있는가. 스킬, MCP, 은행 DB, 국가 오픈 API, LLM 이걸 가지고 있다." (7/3, 김주용)
  > "사실 저희가 계속 얘기했던 게. 무기 에이전트가 무기가 있어야 되는데, 그중에 하나가 이런 공공 API를 가져다 올 수 있는 거다." (7/3, 이승보)

- **4함수계약(computeRiskDecision 등)**: 세 회의 정리본 원문 어디에도 "함수"라는 단어 및 4함수 계약 관련 발화가 없음. **원문 불명확 — 이 회의들에서는 확인 불가.**

- **"LocalGuard"라는 시스템 이름**: 7/3 회의에서 김주용이 "이 시스템의 이름이 뭔가 로컬 가드 그대로 할 건가"라고 미정 상태로 언급했을 뿐, 이승보가 이 이름을 지었거나 확정했다는 원문은 없음(원문 불명확).
  > "그리고 이 시스템의 이름이 뭔가 로컬 가드 그대로 할 건가 이런 것들." (7/3, 김주용)

## 실동작 vs 미완

**실동작/진행 중이라고 말한 것**
- 은행권 하네스: "잘 구동 동작이 되게 하고 있었습니다." (7/2, 이승보)
- 비은행권(JB캐피탈) 하네스: JB캐피탈 사이트 분석 기반으로 동시 작업 중이었으나 일정으로 잠시 중단. (7/2, 이승보)
- 역할별 대시보드: "역할도 제가 작업해서 다 다듬으셨어요." (7/2, 이승보, 김주용의 확인 인용)
- 은행 콘솔(전북은행 축)·캐피탈 콘솔 병행 개발 진행 중. (7/3, 이승보)
- 프롬프트 고도화 시도(정리 화면 캡처 → AI에 정리·고도화 요청). (7/3, 이승보)

**아직 안 됐다고 명시한 것**
- Claude API 실제 호출: "저는 이거 API 아직 호출 안 했거든요. ... 다 같이 있을 때 하려고 아직 안 했습니다." (7/2, 이승보) — 이유는 "뭔가 잘못하면 위험할 것 같아서".
- 로컬 모델(Ollama) 실제 연결 및 구동 시연: 7/3에 "연결해서 ... 구동까지 된다는 내용을 넣으면 무조건 좋을 것 같아요"라고 **제안**하는 형태로만 언급 — 이 시점까지 이승보 쪽에서 실제로 연결·시연했다는 문장은 없음(제안·계획 단계).
- 콘솔 3종(그룹/전북은행/광주은행) → 2종(전북은행/캐피탈) 통합은 7/3 회의에서 막 제안·합의된 상태이며, 실제 반영 완료 여부는 원문에 없음.
- 도메인 개수(4개 vs 3개 vs 5개)·에이전트 개수(14개 vs AI 추천 4~5개)는 7/3 시점까지 미확정. "이것도 저희가 판단하기 힘드네요. 지금은. 그럼 일단 넘어가요. 이거는 AI한테 설계를 한다고 하고." (7/3, 김주용)
- DB 연동 방식(정적/서버/하이브리드): 미정, 승보·주용 별도 논의 예정. (7/3, 김주용)
