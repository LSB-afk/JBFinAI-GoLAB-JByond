# 에이전트 아키텍처 재설계(뽀족한 전문가 로스터)

# JB LocalGuard OS 에이전트 아키텍처 재설계안

## 판정: 코어 5개만으로는 부족하다

**결론:** `RM·여신심사·사후관리·준법·AML` 5개는 **조직 축/메뉴 축**으로는 맞지만, 실제 에이전트 로스터로는 너무 굵다. 본선 설계는 **AI 운영 에이전트 14개 + 사람 승인자 2명 + 승인 게이트**를 유지하는 것이 가장 방어 가능하다. 이 수는 `_canon.md §2`와 일치한다.

근거는 세 가지다.

1. **도메인별 데이터 리스크가 다르다.** 소상공인 여신은 `thin-file·분절 데이터`, 전세는 `시세/공시/실거래/등기 품질 편차`, 사기탐지는 `class imbalance·concept drift`가 핵심이라 하나의 범용 “심사 에이전트”가 처리하면 설명성과 검증성이 약해진다. `D14`
2. **금융 AI는 자동판단보다 보조·승인·감사 구조가 맞다.** 신용정보법/개인정보보호법상 설명·이의제기·재검토 요구와 금융 AI 가이드 흐름상, 에이전트는 최종판정자가 아니라 근거 패킷·초안·검증·상향요청을 만드는 구조여야 한다. `D14`, `_canon §8`
3. **멀티에이전트는 전문화가 있을 때만 정당화된다.** D9는 `검색=grounding`, `Rule Engine=deterministic gate`, `Agent=분해·조정` 구조를 권고하면서도 coordination overhead를 경고한다. 그래서 5개는 부족하고, 20개 이상은 과하다. **14개가 canon과 복잡도 사이의 적정점**이다. `D9`, `_canon §2`

## 권고 구조

**총량:** 운영 AI 에이전트 **14개**, 사람 승인자 **2명**, 승인 게이트 **1개**.

| 층          |  구성 | 역할                                                                              |
| ---------- | --: | ------------------------------------------------------------------------------- |
| 코어 오케스트레이션 |   1 | `Case → AgentRun → Agent → Skill → Evidence → Approval → Audit` 실행 순서·라우팅·중단·재개 |
| 직군별 전문가군   |  10 | RM, 여신심사, 사후관리, AML/피싱, 전세 특화 판단                                                |
| 통제군        |   3 | 준법, 계약, 포트폴리오/감사 지표                                                             |
| 사람 승인      |   2 | `RM 최종 승인자`, `준법 최종 승인자`                                                        |
| 게이트        |   1 | L0~L4 승인 정책 강제, 고객 대상 행동 실행 전 차단                                                |

전세는 **전북은행 전용 도메인**으로 두고, `JB우리캐피탈`에는 전세 라인을 비활성 모듈로 둔다. 캐피탈 확장은 여신·사후관리·피싱/AML 중심으로 붙인다. `사용자 커널`, `_canon §5`

## 확장 방식

새 계열사/도메인은 전체 로스터를 복제하지 않는다. **도메인팩**으로 추가한다.

- `도메인팩 = Lead 1 + Specialist 2~3 + 기존 준법/감사/승인 게이트 재사용`
- 전북은행 전세팩: 이미 `전세위험 관리 리드 + 전세가율 + 등기 권리 + 임차인 손실위험`
- JB우리캐피탈 확장팩: `오토/비오토 여신 사후관리`, `연체 전조`, `담보/차량가치` 등은 `(근거 필요)/(→담당자 확인)`

**심사위원용 한 줄:**  
5개 직군은 메뉴 구조로 쓰고, 실제 운영은 canon 14명의 뾰족한 전문가가 맡는다. 그래야 도메인별 데이터 결함, 금융 규제, 승인·감사 요구를 각각 Evidence와 Audit으로 설명할 수 있다.
