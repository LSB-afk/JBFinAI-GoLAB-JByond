# [프롬프트 원문] 운영 3축 심사방어 — Q13·Q14·Q15 생성 프롬프트

> Q13~Q15 문서와 에이전트 설계도 3종(casesops-분기 08~10)을 도출한 원본 딥프롬프트. 재실행·변형 시 이 원문 기준. 작성: Fable, 2026-07-04. 실행 결과물이 이미 존재하므로 재실행 시 기존 문서와 diff로 대조할 것.

## 프롬프트 원문

**역할**: 너는 금융그룹 CIO 앞에서 AI 에이전트 콘솔 도입을 심사하는 기술실사(Tech DD) 담당자다. 아래 시스템을 상대로 세 가지 공격 질문을 만들고, 그 각각에 대해 **제품 설계로 답하는 방어 논리**를 작성하라.

**시스템 전제 (SSOT — 변경 금지)**:
- 계약: `Case → AgentRun → Agent → Skill → Evidence → Approval → Audit`
- 하이브리드 라우팅: 원본 PII·신용정보는 **로컬 모델만** 처리, 외부 프런티어 LLM은 **Zero-PII 초안 보조만** (`subject_token`, `risk_code`, `score_band`, `evidence_pointer`만 반출)
- 모든 고객 대상 행동은 사람 최종승인 게이트 통과 (금융위 AI 가이드라인: AI는 보조수단)
- 검증된 ROI 주장은 "RM 반복 행정시간 절감 0.83억~7.66억원/년"뿐. 연체 방어·SOM은 상방 시나리오
- 도입비 가정: 초기 5억 + 연 OPEX 1억 `[가정]` / 가격식: `seat fee + 계열사 platform fee`

**공격 질문 3축**:

**A. 유닛 이코노믹스 — "RM 1명이 하루에 태우는 토큰/컴퓨팅 비용이 얼마인가? 절감액보다 운영비가 크면 어떡하나?"**
1. RM 1명의 하루 케이스 처리량 가정(예: 케이스 N건 × AgentRun M회 × 평균 토큰 K)을 세우고, 라우팅 계층별(로컬 소형모델 / 로컬 중형 / 외부 프런티어) 단가로 **RM 1인 월 비용 범위**를 계산하라. 모든 수치에 `[1차]`/`[가정]` 태그를 붙여라.
2. 비용을 구조적으로 낮추는 설계 장치를 우선순위로 제시하라. 후보: ① 작업 티어별 모델 라우팅(분류·추출·요약=로컬 소형, 판단 초안만 프런티어), ② Zero-PII 최소필드 반출이 곧 컨텍스트 절감이라는 이중효과, ③ Evidence 캐시·중복 케이스 디듈리케이션, ④ 야간 배치 vs 실시간 분리, ⑤ seat당 토큰 예산 상한 + 콘솔 내 토큰 미터링(MVP에 이미 토큰 사용 패널 존재 — 이것이 답변의 물증).
3. "비용이 절감액을 역전하는 임계점"(break-even 케이스 단가)을 명시하고, 그 아래로 유지하는 운영 규칙을 제안하라.

**B. 오류 관측가능성 — "이 복잡한 라우팅에서 오류가 나면 어떻게 알고, 어떻게 복구하나?"**
1. 실패 지점을 계약 단계별로 열거하라(수집 실패 / 라우팅 오선택 / 로컬모델 타임아웃 / 외부 API 한도 / 검증 불일치 / 승인 후 실행 실패).
2. 각 실패 지점에 대해 `AgentRun` 단위 구조화 로그 스키마(run_id, case_id, model_tier, latency, token_in/out, error_class, retry_count, fallback_path)를 정의하고, **폴백 사다리**(프런티어 실패→로컬 재시도→사람 큐 격상)를 설계하라.
3. "AI가 조용히 틀리는" 경우(오류 없이 나쁜 출력)는 Verification Agent 불일치율 + 사람 반려율을 드리프트 지표로 삼는 논리를 세워라.

**C. 감사 로그 실효성 — "로그를 쌓는 건 알겠는데, 그게 정말 쓸모 있나? 누가 언제 왜 보나?"**
1. 감사 로그를 **소비자별로** 분해하라: ① 금융당국(사람 최종승인 증적·PII 비반출 증명) ② 내부 감사·분쟁(왜 이 판단을 했는지 Evidence 체인 재생) ③ 운영자(폴백·오류 패턴) ④ 재무(토큰 원가 정산) ⑤ 모델 개선(반려 사유 데이터셋). **소비자가 없는 로그는 남기지 않는다**는 원칙으로 각 로그의 보존기간·접근권한을 표로 만들어라.
2. "감사 로그 자체가 PII 저장소가 되는 역설"을 어떻게 피하는지(포인터 방식 — 로그에는 evidence_pointer만, 원본은 계열사 경계 안) 답하라.

**출력 형식**: 각 축마다 ① 심사위원 예상 질문 원문(구어체) ② 30초 구두 답변(발표자용) ③ 근거 상세(근거팩 형식, `[1차]/[가정]/[미검증]` 태그) ④ 우리 MVP에서 지금 보여줄 수 있는 물증(화면/코드). 금지: 검증 안 된 수치를 확정처럼 말하기.
