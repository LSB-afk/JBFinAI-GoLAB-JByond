# [Q14] 라우팅 오류 관측·복구 — 구조화 로그와 폴백 사다리

## 예상 질문 (구어체)

"모델을 이리저리 라우팅하다 오류 나면 어떻게 압니까? 로그는 어떻게 남기고, 복구는 누가 합니까?"

## 30초 구두 답변

"실패를 케이스 계약 단계별로 관측합니다. 모든 호출은 AgentRun 단위 구조화 로그 한 줄 — run_id, case_id, 엔진, 티어, 지연, 토큰, 오류 클래스, 재시도 횟수, 폴백 경로 — 로 남고, 복구는 폴백 사다리가 자동으로 탑니다: 요청 엔진 실패 → 반대 엔진 재시도 → 둘 다 실패하면 사람 큐로 격상됩니다. AI가 조용히 사라지는 게 아니라 반드시 사람 앞에 도착하는 구조입니다. 지금 데모에서 타임아웃을 일부러 내면 폴백이 실제로 작동하는 걸 보여드릴 수 있습니다."

## 근거 요점

- 오류 분류 체계(구현됨): `timeout / engine_missing / nonzero_exit / parse_error / empty_output / all_engines_failed`.
- 폴백 사다리(구현됨): claude ↔ codex 상호 폴백 1회 → 실패 시 `escalated: true` + `queue: "human"` 응답. 실패 시도도 각각 로그 1줄(재시도 원가까지 관측).
- "조용히 틀리는" 경우(오류 없이 나쁜 출력)는 별도 축: Verification Agent 불일치율 + 사람 반려율을 드리프트 지표로 사용 `[설계, 미구현]`.
- 로그 스키마: `{runId, caseId, ts, engine, tier, latencyMs, tokensIn, tokensOut, costUsd, errorClass, retry, fallbackPath, escalated}` — 서버 이식 시 그대로 테이블 DDL이 된다.

## MVP 물증

- `02_제품/scripts/api-proxy.mjs` `handleLlm()` — 폴백 사다리 + 시도별 JSONL 기록.
- 토큰 패널 실측 블록 — "성공 n회 · 오류 n회 · 사람 큐 격상 n회" 카운터 표시.
- 시연법: 프록시 없이 호출 → 앱 폴백 / `timeoutMs: 1`로 호출 → 사다리 작동 후 escalated 응답.

## 남은 것

- escalated 건을 콘솔 승인 큐에 수동처리 항목으로 노출하는 UI 연결 `[미구현, 게이트 1 후속]`.
