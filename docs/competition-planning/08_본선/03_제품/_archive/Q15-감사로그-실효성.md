# [Q15] 감사 로그 실효성 — 누가, 언제, 왜 보는가

## 예상 질문 (구어체)

"로그 많이 쌓는 건 알겠는데, 그거 진짜 쓸모 있습니까? 누가 언제 봅니까? 로그 창고만 커지는 거 아닌가요?"

## 30초 구두 답변

"원칙이 하나 있습니다 — 소비자 없는 로그는 남기지 않는다. 저희 감사 레코드에는 용도 태그가 붙어 있어서, 이 로그를 누가 소비하는지 화면에서 바로 보입니다. 당국 증적은 사람 최종승인 이력, 분쟁 재생은 판단 근거 체인 재구성, 운영 점검은 오류·폴백 패턴, 원가 정산은 토큰 비용입니다. 그리고 감사 로그 자체가 PII 창고가 되지 않도록 로그에는 evidence 포인터만 남기고 원본은 계열사 경계 안에 둡니다. 해시 체인 무결성 검증과 JSON 내보내기도 콘솔에서 바로 시연됩니다."

## 근거 요점 — 소비자 × 용도 매트릭스

| 용도 태그 | 소비자 | 무엇에 쓰나 | 근거 소스 |
|---|---|---|---|
| 당국 증적 | 금융당국·준법 | 사람 최종승인 이력, PII 비반출 증명 | 금융위 AI 가이드라인(AI=보조수단) |
| 분쟁 재생 | 내부 감사·법무 | "왜 이 판단을 했나" Evidence 체인 재구성 | Case→Evidence→Approval 계약 |
| 운영 점검 | 운영자 | 오류 클래스·폴백 경로·드리프트 패턴 | Q14 로그 스키마 |
| 원가 정산 | 재무 | 케이스·티어별 토큰 원가 | Q13 원장(llm-runs.jsonl) |

- 보존기간·접근권한 분리는 서버 이식 시 정책 테이블로 `[설계, 미구현]`.
- PII 역설 방어: 감사 레코드에는 `evidenceId`(포인터)만 — 원문은 계열사 경계 안. 게이트웨이도 cwd 중립화로 볼트 컨텍스트 유입 차단(구현됨).

## MVP 물증

- `02_제품/app/app.js` `auditPurpose()` + `auditChainRecords()` — 레코드별 용도 태그, 감사 로그 화면에 `용도 ○○` 표시.
- 기존 물증: 해시 체인(`prev → hash`), 무결성 검증 버튼, JSON 내보내기.

## 남은 것

- 용도 태그를 정규식 추론이 아닌 이벤트 발생 시점의 명시 필드로 승격(서버 이식 시).
