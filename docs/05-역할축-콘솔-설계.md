# 역할축 AI 업무 콘솔 설계 (2026-07-03)

## 최종 설계 한 줄

이 프로젝트는 "도메인별 플랫폼"이 아니라 **역할별 AI 업무 콘솔**이다.
**역할이 콘솔을 결정하고, 도메인은 그 역할이 처리하는 케이스가 되며,
에이전트는 케이스를 처리하는 내부 전문가 조직으로 동작한다.**

## 3축 분리

- **역할 축(누가)**: RM · 기업여신 담당자 · 전세사기 보호 담당자 · FDS/보이스피싱 담당자
- **도메인 축(무엇을)**: 여신 심사 · 전세 위험 점검 · 보이스피싱 대응 · JB우리캐피탈 운영
- **계열사 축(어디서)**: 전북은행 · JB우리캐피탈 (광주은행은 현 UI에서 제외)

화면 구조: 계열사 선택 → 역할 선택 → 역할 전용 하네스 → 그 역할의 도메인 케이스 보드.

## 실동작 콘솔 4개 (A안)

| 콘솔 | roleKey / route | 도메인·대표 케이스 | 금지(훅으로 강제) |
| --- | --- | --- | --- |
| 기업여신 심사지원 포털 | `corporate-credit` · `#/roles/corporate-credit` | 소상공인 대출 검토 — 전주 카페(CCL-0001) | 대출 승인/거절·금리/한도·신용등급 확정 |
| 전세사기 보호 업무지원 포털 | `jeonse-protection` · `#/roles/jeonse-protection` | 전세 위험 점검 — 신축빌라(JEONSE-0001) | 전세사기 확정·법률/보증/피해자 결정 확정 |
| FDS·보이스피싱 대응 포털 | `fds-response` · `#/roles/fds-response` | 고위험 이체/고령 고객(FDSC-0001) | 자동 종결·차단/지급정지 실행·거래 실행 |
| JB우리캐피탈 운영지원 포털 | `jb-woori-capital` | 계열사 확장성 증명(자동차/개인금융/고객보호/FDS) | 계열사 공통 정책 |

## 에이전트 구조: 표면 ↔ 내부

- 표면 메뉴 5~6개 / 내부 agent registry 8~10개 / 시연 시 케이스당 활성 3~5개.
- 하네스 화면은 "표면 에이전트" 카드와 "내부 전문 조직(접힘)"을 분리 표시하고,
  전세보호 콘솔은 케이스별 활성 에이전트(runs 기준)를 함께 보여준다.

## 공통 플랫폼 뼈대 (모든 콘솔 공유)

Case → Agent → Skill → Evidence → Audit → PII Guard
(구현: harnessCore 훅 엔진 + harnessRegistry manifest + harnessVerification 자체 검증 6종 × 4콘솔 = 24 체크)

## 역할별로 분리한 것 (공유 금지)

route · sidebar menu · case lifecycle · domain taxonomy · search · count query ·
case creation flow · agent registry · risk rules · seed data · dashboard KPI · audit/escalation flow.
접두 네임스페이스: `jpo`(전세) · `ccl`(기업여신) · `fdr`(FDS) · `jbwc`(캐피탈), 저장소도 각자
(`jpo-ops-db-v2` / `ccl-ops-db-v1` / `fdr-ops-db-v1` / `jbwc-ops-db-v3`).

## 참고

- RM은 역할 레일에 유지되나 현재는 요약 대시보드(로드맵) — 콘솔 승격 후보.
- label-only였던 `corporate-credit-dashboard`/`fds-dashboard`는 라우팅에서 제거되었고,
  verify(`#…-dashboard` 해시 재유입 금지)와 registry `forbiddenResurface`가 재유입을 차단한다.
